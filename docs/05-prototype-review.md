# EC 工单系统 HTML 原型固化与验收记录

## 1. 原型版本说明

| 项目 | 内容 |
|------|------|
| 原型版本 | v0.1 |
| 原型用途 | 客户流程确认版 |
| 原型范围 | MVP 静态 HTML 原型 |
| 页面范围 | 7 个页面 |
| 角色范围 | 4 类角色 |
| 状态范围 | 7 个状态 |
| 工单类型范围 | 3 类工单 |
| 当前状态 | 待客户确认 |

本次固化只围绕现有 `prototype/index.html` 进行原型验收、截图归档、问题清单整理和允许范围内的小修，不新增页面、不新增角色、不新增状态、不新增二期功能。

本次小修内容：

1. 修正顶部主流程文案，使其与 MVP 主流程保持一致：CS 建单 / 来电记录 -> CS 受理 -> Team Leader 分派 -> 师傅接单 / 拒单 -> 师傅处理并上传记录 -> Team Leader 或主管确认 -> 工单关闭 -> 基础统计。
2. 优化手机尺寸下师傅角色的分派与接单视图排序，使师傅待接单任务优先展示，Team Leader 分派区仍保留在页面下方用于流程确认。

## 4. 问题清单

### 4.1 必须修复

暂无必须修复问题。

本次发现的两个小问题已处理：

1. 顶部主流程文案不完整，已修正。
2. 手机师傅端分派页首屏不够聚焦，已调整为师傅任务优先展示。

### 4.2 建议优化

| 问题 | 影响 | 建议 |
|------|------|------|
| 分派与接单页面仍同时展示 Team Leader 视图和师傅视图 | 有利于客户理解完整流程，但真实系统中不同角色可能不应看到对方操作区。 | 后续开发时按真实权限隐藏非本角色操作区；当前原型保留用于流程确认。 |
| 关键词筛选为实时过滤 | 原型可用，但后续数据量较大时可能需要更稳的交互。 | 后续开发可改为回车搜索、点击查询或防抖搜索。 |
| 统计报表仅为轻量条形图 | 已满足 MVP 流程确认，但不是最终报表体验。 | 客户确认统计口径后，再决定是否增强图表、导出或钻取。 |

### 4.3 可客户确认后再处理

| 问题 | 当前处理 | 是否阻塞客户确认 |
|------|----------|------------------|
| PM / CM 的最终中文定义 | 原型使用 PM / 维修 / 保养、CM / 特殊任务。 | 不阻塞，但客户需确认命名。 |
| 通知渠道到底用系统内、钉钉、WhatsApp 还是邮件 | 原型只写内部通知节点，不做外部自动通知。 | 不阻塞原型确认，阻塞后续开发细化。 |
| 附件 / 照片是否必填 | 原型展示上传字段，但未强制必填。 | 不阻塞原型确认。 |
| 已关闭后是否允许重开 | 原型写为待客户确认，不提供重开按钮。 | 不阻塞 MVP 主流程确认。 |
| 已取消是否进入统计 | 原型已纳入状态统计。 | 需客户确认统计口径。 |
| 师傅端是否必须移动端优先 | 原型已提供手机视口截图。 | 不阻塞流程确认，影响后续开发优先级。 |

### 4.4 不进入 MVP

以下内容不进入当前 v0.1 客户确认版原型主流程：

- 客户自助门户。
- 投诉独立工作台。
- 政府 / 合约独立流程。
- 报价 / 采购 / 安装完整流程。
- 报销 / 考勤流程。
- WhatsApp 自动通知。
- SLA（Service Level Agreement，服务级别协议）超时升级。
- AI 图片检测。
- ERP / CRM 深度集成。
- 成本、绩效、财务、采购等复杂报表。

## 6. 固化结论

当前 `prototype/index.html` 可以作为 EC 工单系统 MVP 静态 HTML 原型 v0.1 客户流程确认版。

进入客户确认前，建议重点让客户确认以下 5 件事：

1. 是否认可当前 1 条主流程。
2. 是否认可 3 类 MVP 工单类型。
3. 是否认可 4 类 MVP 角色与职责。
4. 是否认可 7 个工单状态和拒单回退规则。
5. 是否认可通知、附件、重开、取消统计和移动端优先级的当前处理方式。
