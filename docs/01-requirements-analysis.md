# EC 工单系统需求规格自检与范围一致性分析

**版本**: 1.0.0  
**日期**: 2026-06-12  
**阶段**: 需求规格自检与范围一致性分析；不新增需求、不写代码、不生成 UI（User Interface，用户界面）、不做数据库或接口设计。  
**分析范围**: `.specify/memory/constitution.md`、`docs/01-requirements-spec.md`、`docs/01-requirements-clarification.md`、`materials` 文件夹客户资料。

## 1. 总体结论

当前 `docs/01-requirements-spec.md` 不建议直接作为 HTML 原型输入，但可以在完成 MVP 收敛后进入流程图阶段。

核心判断：

| 检查项 | 结论 | 说明 |
|--------|------|------|
| 是否可进入流程图阶段 | 有条件可以 | 必须以 `docs/01-requirements-clarification.md` 的收敛口径为准，先将 `docs/01-requirements-spec.md` 中过宽内容调整清楚。 |
| 是否可直接进入 HTML 原型 | 暂不建议 | 原规格仍包含过多角色、工单类型、状态和二期功能，直接做原型容易变成多模块系统。 |
| 是否符合项目宪法 | 部分符合 | 需求追溯较完整，但 MVP 优先、状态权限一致、一线可用性仍需进一步收敛。 |
| 是否存在范围膨胀风险 | 存在 | 投诉、政府/合约、报价/采购/安装、报销/考勤、客户自助、CR/采购/财务/HR 工作台等内容仍容易被误读为 MVP。 |
| 主流程是否闭环 | 基本闭环 | 建单、受理、分派、接单/拒单、处理、确认、关闭、统计均已覆盖，但状态和角色需要按澄清文档简化。 |

建议结论：先更新 `docs/01-requirements-spec.md`，再生成 `docs/02-ticket-flow-analysis.md`。流程图可以开始，但必须以“1 条主流程、3 类工单、4 类角色、7 个状态”为边界。

## 2. MVP 范围检查

### 2.1 应保留在 MVP 的内容

| 内容 | 保留原因 | 对应资料/依据 |
|------|----------|---------------|
| CS 建单 / 来电记录 | 客户资料明确存在来电、WhatsApp、现场沟通等入口，需要 CS 人工记录。 | M1、M5 |
| CS 受理 | 需要判断问题是否有效、是否现场解决、是否需要派工。 | M1 |
| Team Leader 分派 | 客户资料明确涉及按区域、团队、技能和人工判断分派。 | M1、M2 |
| 师傅接单 / 拒单 | 客户资料明确提到师傅可接单或拒单，并需记录拒单原因。 | M1、M2 |
| 师傅处理记录 | 需要记录检查、维修、未完成原因、完成说明等。 | M2、M3 |
| 附件 / 图片上传 | 现场照片、PDF、服务报告、完工单等资料在客户材料中多次出现。 | M1、M2、P |
| Team Leader 或主管确认 | 需要有人确认处理结果后关闭，避免流程不闭环。 | M2、P；确认人需客户确认 |
| 工单关闭 | 作为流程结束点，支撑统计和归档。 | M2、P |
| 基础查询 | 支持按状态、客户、区域、负责人、时间等查看。 | M2、M3 |
| 基础统计 | 支持查看待处理、处理中、已完成/已关闭、按类型/处理人/区域等统计。 | M1、M2 |
| 基础角色权限 | 保证 CS、Team Leader、师傅、主管看到和操作的内容不同。 | 项目宪法、M1、M2 |

### 2.2 应降级为字段 / 标签的内容

| 内容 | 建议处理 | 原因 |
|------|----------|------|
| 投诉 / 客户关系 | 作为工单分类、标签、备注或关联跟进人字段。 | 独立投诉流程、CR 权限、升级规则资料不足。 |
| 政府 / 合约 | 作为客户类型、项目类型、编号前缀、附件来源或备注字段。 | 政府项目存在特殊性，但细则不足，不应形成独立 MVP 流程。 |
| 报价 / 采购 / 安装关联 | 仅保留报价编号、PO、ML、安装记录等关联字段。 | 完整报价采购安装链路过长，涉及销售、采购、仓库、财务。 |
| 报销 / 考勤关联 | 可在处理记录中备注，不做审批流。 | 属于 HR、财务或报销系统范围，不是工单 MVP 主流程。 |
| 客户自助查询 | 先作为二期候选，不做 MVP 独立入口。 | 外部客户入口、身份校验、可见范围未确认。 |

### 2.3 应放入二期的内容

| 内容 | 二期原因 |
|------|----------|
| 客户自助提交 / 查询 / 扫码查进度 | 需要外部访问、权限控制、客户身份确认和展示范围设计。 |
| 投诉 / CR 独立工作台 | 需要明确投诉分类、升级规则、CR 职责和客户反馈闭环。 |
| 政府 / 合约独立流程 | 需要明确编号、派工、交付文件、验收规则和特殊流程。 |
| 完整报价 / 采购 / 仓库 / 安装流程 | 涉及 PO、ML、到货、库存、安装、验收、财务等多角色。 |
| 报销 OCR（Optical Character Recognition，光学字符识别） | 与票据、费用、财务审核相关，不属于工单主流程。 |
| 考勤 / 薪酬 / 绩效 | 属于 HR（Human Resources，人力资源）系统或独立管理模块。 |
| 自动客服 | 属于自动化服务能力，需要另行确认场景和数据来源。 |
| AI（Artificial Intelligence，人工智能）图片质量检测 / 地址检索 | 属于增强能力，不影响 MVP 闭环。 |
| 多系统深度集成 | 会显著增加实施风险，应在核心流程稳定后再评估。 |

### 2.4 应明确不做的内容

| 内容 | 判断 |
|------|------|
| 真实后端开发 | 当前阶段不做。 |
| 数据库实现 | 当前阶段不做。 |
| 接口设计或接口开发 | 当前阶段不做。 |
| 复杂 ITSM（IT Service Management，信息技术服务管理） | 不进入 MVP。 |
| 资产管理 | 不进入 MVP。 |
| 知识库 | 不进入 MVP。 |
| SLA（Service Level Agreement，服务级别协议）计费 | 不进入 MVP。 |
| 实时定位监控 | 客户资料中已明确不做实时监控方向，不进入 MVP。 |
| WhatsApp 截图自动抓取 | 不进入 MVP。 |
| 语音转写自动建单 | 不进入 MVP。 |

## 3. 主流程闭环检查

目标主流程：

CS 建单 / 来电记录  
→ CS 受理  
→ Team Leader 分派  
→ 师傅接单 / 拒单  
→ 师傅处理并上传记录  
→ Team Leader 或主管确认  
→ 工单关闭  
→ 基础统计

| 流程节点 | 当前规格覆盖情况 | 是否闭环 | 分析 |
|----------|------------------|----------|------|
| CS 建单 / 来电记录 | 已覆盖 | 是 | `docs/01-requirements-spec.md` 已明确电话、WhatsApp、现场沟通等入口。 |
| CS 受理 | 已覆盖 | 是 | 已包含判断现场解决、需派工、需补充资料或取消。 |
| Team Leader 分派 | 已覆盖 | 是 | 已包含区域、团队、技能、人工判断分派。需进一步确认分派规则优先级。 |
| 师傅接单 / 拒单 | 已覆盖 | 是 | 已包含接单/拒单和拒单原因。建议拒单作为动作，不作为主状态。 |
| 师傅处理并上传记录 | 已覆盖 | 是 | 已包含处理记录、照片/附件、完成说明和异常原因。 |
| Team Leader 或主管确认 | 部分覆盖 | 基本闭环 | 规格中存在“负责人/客户确认”等表述，需统一为 MVP 默认 Team Leader 或主管确认。 |
| 工单关闭 | 已覆盖 | 是 | 已有关闭节点，但关闭后是否允许重开需确认。 |
| 基础统计 | 已覆盖 | 是 | 已有状态、类型、处理人、区域、未完成事项等统计，但指标应按 7 个 MVP 状态重新整理。 |

主流程缺口：

1. “现场解决”路径需要明确进入“已关闭”还是“已取消”。建议现场解决进入“已关闭”，无效/重复/客户取消进入“已取消”。
2. “客户确认”不应作为 MVP 默认必经节点。若客户未确认，则由 Team Leader 或主管确认关闭。
3. “报价/采购/材料不足”应作为异常原因或关联字段，不应在 MVP 中拉出“待报价/待采购/待安装”独立流程。
4. “基础统计”应明确从关闭和状态流转自动汇总，不需要单独人工填报流程。

## 4. 状态机检查

建议 MVP 状态：

1. 待受理
2. 待分派
3. 待接单
4. 处理中
5. 待确认
6. 已关闭
7. 已取消

### 4.1 状态是否够用

这 7 个状态足够支撑 MVP 主流程。相比当前 `docs/01-requirements-spec.md` 中的草稿、已受理、已分派、已接受、已拒绝、待客户补充、待客户确认、已完成、已驳回等状态，7 状态模型更适合第一版：

| 状态 | 覆盖的业务含义 |
|------|----------------|
| 待受理 | 覆盖提交后等待 CS 判断，也可承接来电记录。 |
| 待分派 | 覆盖已受理但未分派，也承接拒单后的重新分派。 |
| 待接单 | 覆盖已分派但师傅未接单。 |
| 处理中 | 覆盖师傅已接单、现场处理、阶段性跟进、异常等待。 |
| 待确认 | 覆盖师傅提交完成，等待 Team Leader 或主管确认。 |
| 已关闭 | 覆盖完成后归档和统计。 |
| 已取消 | 覆盖无效、重复、客户取消、无需处理。 |

### 4.2 重点检查结论

| 检查点 | 结论 | 建议 |
|--------|------|------|
| 拒单后是否回到待分派 | 应回到待分派 | 拒单作为动作记录，必须填写拒单原因，并通知 Team Leader。 |
| 取消由谁操作 | 需客户确认 | 建议 CS 可在受理阶段取消；Team Leader 或主管可在分派/处理中取消；师傅不直接取消工单，只能反馈无法处理。 |
| 待确认由谁确认 | 需统一 | MVP 建议由 Team Leader 或主管确认，不默认客户确认。 |
| 已关闭后是否允许重开 | 需客户确认 | MVP 可先不做重开；如必须处理，建议作为“重开动作”记录原因，而不是新增主状态。 |
| 是否需要草稿 | 非必需 | CS 可在页面未提交前暂存，但不进入正式状态机。 |
| 是否需要待客户补充 | MVP 不建议 | 先作为处理记录或异常原因，客户确认后再考虑独立状态。 |
| 是否需要已完成 | MVP 不建议 | “待确认 → 已关闭”已经能表达闭环，避免“已完成”和“已关闭”口径混淆。 |

## 5. 角色权限检查

### 5.1 四类 MVP 角色是否足够

四类角色足以支撑 MVP：

| 角色 | 支撑的流程节点 | 是否足够 |
|------|----------------|----------|
| CS / 工单受理人 | 建单、来电记录、受理、补充信息、标记取消或现场解决。 | 足够 |
| Team Leader / 分派与确认人 | 分派、重新分派、接收拒单反馈、确认处理结果。 | 足够 |
| 师傅 / 处理人 | 接单、拒单、处理、上传照片/附件、提交完成。 | 足够 |
| 主管 / 管理员 | 查看全局、处理异常、关闭/确认、基础配置和统计。 | 足够 |

### 5.2 非 MVP 独立工作台角色影响

| 角色 | 不作为独立工作台是否影响 MVP | 建议处理 |
|------|------------------------------|----------|
| 客户 / 提交人 | 不影响 | 由 CS 代为建单；客户作为联系人、公司、来电人、联系方式字段。 |
| CR / 客户关系 | 不影响主流程 | 投诉/客户关系作为标签或备注，CR 不做独立工作台。 |
| 采购 / 仓库 | 不影响主流程 | PO、ML、材料、到货等作为关联字段或备注，完整流程二期。 |
| 财务 | 不影响主流程 | 报价、发票、报销作为关联字段或二期模块。 |
| HR | 不影响 | 考勤、薪酬、绩效不进入工单 MVP。 |

### 5.3 当前权限矩阵问题

当前 `docs/01-requirements-spec.md` 的权限矩阵包含客户、CS、CR、Team Leader、师傅、主管、采购、财务 8 类角色，作为需求全集可以保留，但不适合作为 MVP 原型依据。

主要风险：

1. 客户“查看/反馈/上传附件”容易被误解为客户自助入口进入 MVP。
2. CR、采购、财务权限会导致原型需要额外工作台。
3. CS 与 Team Leader 的分派权限存在重叠，需确认谁是主要分派人。
4. 主管可关闭、Team Leader 可确认完成，需明确关闭动作是否两者都可操作。

## 6. 页面原型准备度

如果按澄清后的 MVP 边界，后续 HTML 原型可以先做以下 7 个页面；但不建议在更新 `docs/01-requirements-spec.md` 前直接制作高保真页面。

| 页面 | 是否可先做 | 对应流程节点 | 原型重点 |
|------|------------|--------------|----------|
| 工单看板 / 首页 | 可以 | 基础统计、异常提醒、待处理入口 | 展示待受理、待分派、待接单、处理中、待确认、已关闭、已取消数量。 |
| 工单列表 | 可以 | 查询、筛选、进入详情 | 按状态、类型、区域、处理人、时间筛选；只展示 MVP 三类工单。 |
| 新建工单 / 来电记录 | 可以 | CS 建单 / 来电记录、CS 受理 | 客户、联系人、电话、区域、问题描述、工单类型、附件、是否需派工。 |
| 工单详情 / 处理记录 | 可以 | 全流程查看、处理记录、附件、状态历史 | 展示基础信息、状态、责任人、处理记录、附件、拒单原因、关闭信息。 |
| 分派与接单视图 | 可以 | Team Leader 分派、师傅接单 / 拒单 | 分派师傅、查看拒单原因、师傅接单/拒单、重新分派。 |
| 统计报表 | 可以 | 基础统计 | 只做 MVP 基础统计，不做复杂经营、成本、绩效或 AI 报表。 |
| 基础设置，简化版 | 可以，需简化 | 支撑类型、区域、角色、附件规则 | 只做类型、区域、角色、是否允许拍照等基础口径展示，不做完整系统配置。 |

页面准备度结论：

1. 页面清单基本覆盖 MVP 主流程。
2. 原规格中的“工单受理页、工单分派页、待确认/关闭页”可以在原型中作为详情页内的角色视图或操作区，不一定拆成多个独立页面。
3. 不建议制作客户门户、CR 工作台、采购/仓库工作台、财务工作台、HR 工作台。
4. 师傅端是否需要移动端优先展示，必须在原型前确认。

## 7. 风险与待确认

进入流程图和原型前最需要确认的 Top 10 问题：

| 优先级 | 问题 | 风险 |
|--------|------|------|
| 1 | MVP 是否确认只做内部工单闭环，不做客户自助提交/查询？ | 影响是否需要客户门户和外部权限。 |
| 2 | MVP 工单类型是否只保留来电/服务记录、PM/维修/保养、CM/特殊任务？ | 影响流程图、表单、筛选、统计口径。 |
| 3 | PM 与 CM 的准确业务定义和中文名称是什么？ | 影响客户理解和一线录入准确性。 |
| 4 | 分派主责是谁：CS、Team Leader，还是两者都可以？ | 影响权限、按钮和状态流转。 |
| 5 | 分派规则优先级是什么：区域、团队、技能、空闲情况、人工判断？ | 影响分派页面和后续可研。 |
| 6 | 师傅接单/拒单是否必须在手机端完成？ | 影响 HTML 原型布局和交互重点。 |
| 7 | 工单完成后由 Team Leader、主管、CS 还是客户确认？ | 影响“待确认”状态和关闭权限。 |
| 8 | 哪些附件/照片必须上传，哪些场景允许不拍照？ | 影响处理页必填项和验收规则。 |
| 9 | MVP 通知渠道是什么，是否必须使用钉钉？ | 影响通知节点说明和后续技术可研。 |
| 10 | 已关闭工单是否允许重开或补充记录？ | 影响状态机和审计记录。 |

## 8. 修改建议

建议修改 `docs/01-requirements-spec.md` 的以下章节：

| 章节 | 修改建议 | 修改原因 |
|------|----------|----------|
| 3. 建设目标 | 聚焦内部工单闭环，客户自助、报价采购、报销考勤改为二期或关联说明。 | 当前目标仍覆盖过多方向。 |
| 4. 用户角色与职责 | MVP 角色只保留 CS、Team Leader、师傅、主管/管理员；其他角色移到二期或字段说明。 | 避免原型出现过多工作台。 |
| 5. 工单类型 | MVP 只保留三类工单；投诉、政府、报价采购、报销考勤、客户自助降级。 | 当前 8 类工单不利于 MVP 闭环。 |
| 6. 工单主流程 | 改成单一主流程，并补充现场解决、取消、拒单退回、确认关闭的路径。 | 让流程图可以直接落地。 |
| 7. 工单状态机 | 改为 7 个 MVP 主状态；删除或降级草稿、已受理、已分派、已接受、已拒绝、待客户补充、待客户确认、已完成、已驳回。 | 当前状态机过复杂。 |
| 8. 功能模块清单 | MVP 模块仅保留建单、列表、详情、受理、分派、接单/拒单、处理记录、附件、确认关闭、查询、统计、基础权限。 | 防止二期功能混入 MVP。 |
| 9. 页面清单 | 页面改为 7 个原型页面：看板、列表、新建/来电记录、详情/处理记录、分派与接单、统计、简化设置。 | 更贴合流程原型，不做多系统页面。 |
| 10. 字段清单 | 将字段分为 MVP 必填、MVP 可选、标签/关联字段、二期字段、待确认字段。 | 当前字段数量偏多，部分会被误解为必做。 |
| 11. 权限矩阵 | 改为四角色权限矩阵，并单独说明非 MVP 角色如何以字段/备注出现。 | 减少权限冲突。 |
