# EC 工单系统需求澄清与 MVP 收敛

**版本**: 1.0.0  
**日期**: 2026-06-12  
**阶段**: 需求澄清、MVP（Minimum Viable Product，最小可行版本）收敛与客户确认问题整理；不写代码、不生成 UI（User Interface，用户界面）、不做数据库或接口设计。

## 资料依据

本文档基于以下资料进行收敛判断：

- `.specify/memory/constitution.md`：EC 工单系统项目宪法，尤其是需求追溯、MVP 优先、流程设计、状态权限一致和中文交付原则。
- `docs/01-requirements-spec.md`：当前需求规格文档。
- `materials/听记纪要/`：客户多次沟通听记。
- `materials/纸质材料图片/`：现有纸质流程、服务报告、报价单、发票、完工单、安装/维修记录、排班/项目清单等资料。

文中沿用 `docs/01-requirements-spec.md` 的资料标签：M1-M7、P。资料未明确的内容均标注为“合理推断，需客户确认”或“待客户确认”。

## 1. 澄清结论

当前 `docs/01-requirements-spec.md` 已经具备继续推进的基础，因为它完整整理了客户业务背景、角色、工单类型、主流程、状态、字段、权限、通知和统计需求，并保留了来源追溯。

但从 MVP 收敛角度看，当前规格仍然偏宽，需要在进入流程图和 HTML 原型前先收敛。主要原因如下：

| 检查项 | 当前判断 | 澄清结论 |
|--------|----------|----------|
| MVP 范围是否过大 | 偏大 | 当前规格把投诉、政府/合约、报价/采购/安装、报销/考勤、客户自助等都纳入了需求视野，容易把第一版扩大为多模块系统。 |
| 工单类型是否过多 | 偏多 | 当前列出 8 类工单，其中多类更适合作为标签、关联字段或二期模块，不应在 MVP 中形成独立复杂流程。 |
| 角色是否过多 | 偏多 | 当前包含客户、CS、CR、Team Leader、师傅、主管、采购、财务、HR 等角色。MVP 原型若全部做独立工作台，会增加确认和开发成本。 |
| 状态机是否过复杂 | 偏复杂 | 当前包含草稿、已受理、已分派、已接受、已拒绝、待客户补充、待客户确认、已驳回等细分状态，超过第一版闭环需要。 |
| 明确资料支持 | 较充分 | CS 来电记录、区域/团队分派、师傅接单/拒单、处理记录、照片/附件、完成确认、基础统计均有资料支撑。 |
| 合理推断内容 | 较多 | 客户自助入口、客户确认关闭、部分通知渠道、优先级、复杂查询、完整权限矩阵等仍需客户确认。 |
| 必须客户确认内容 | 明确 | 工单入口、PM/CM 口径、分派规则、通知渠道、关闭确认人、附件要求、移动端范围等是原型前的关键确认项。 |
| 应放入二期内容 | 明确 | 复杂采购/仓库、报销 OCR、HR/薪酬/绩效、客户自助查询、自动客服、AI 分析、多系统深度集成应放入二期。 |
| 不应进入第一版内容 | 明确 | 实时定位监控、语音转写、WhatsApp 自动抓图、复杂 ITSM、知识库、资产管理、SLA 计费不进入 MVP。 |
| HTML 原型落地风险 | 存在 | 如果不先收敛角色、状态和工单类型，HTML 原型会变成多系统拼盘，客户难以确认主流程。 |

澄清结论：需求规格可以继续推进，但必须先完成 MVP 收敛。建议先更新 `docs/01-requirements-spec.md`，再生成 `docs/02-ticket-flow-analysis.md`，暂不直接进入 HTML 原型。

## 2. MVP 收敛建议

### 2.1 MVP 主流程

MVP 主流程建议收敛为一条内部工单闭环：

CS（Customer Service，客服/工单受理人）建单 / 来电记录  
→ CS 受理  
→ Team Leader（团队负责人）分派  
→ 师傅接单 / 拒单  
→ 师傅处理并上传记录  
→ Team Leader 或主管确认  
→ 工单关闭  
→ 基础统计

该主流程直接对应客户资料中反复出现的来电记录、区域/团队分派、师傅执行、上传照片/记录、负责人确认和管理统计诉求。来源：M1、M2、M5、P。

### 2.2 MVP 工单类型

MVP 工单类型建议只保留以下 3 类：

| MVP 工单类型 | 定位 | MVP 处理方式 | 来源依据 |
|--------------|------|--------------|----------|
| 来电/服务记录工单 | CS 记录客户电话、WhatsApp、现场沟通等请求，并判断是否需要派工。 | 作为工单入口和受理前记录，可转为 PM 或 CM，也可记录为已解决/已取消。 | M1 |
| PM/维修/保养类工单 | 常规维修、保养或可归入标准现场服务的任务。 | 进入标准派工、接单、处理、确认、关闭流程。 | M1、M2 |
| CM/特殊任务类工单 | 无固定执行人、需按技能/空闲情况临时分派的特殊任务。 | 使用同一主流程，仅在类型、技能要求、备注字段上区分，不单独做复杂流程。 | M1、M2 |

说明：PM 和 CM 的最终中文名称、业务定义、是否覆盖“维修/保养/特殊任务”仍需客户确认。

### 2.3 MVP 中只作为分类、标签或关联字段的内容

以下内容不建议在 MVP 中作为独立复杂流程，仅作为分类、标签、备注或关联字段保留：

| 内容 | MVP 处理方式 | 不进入独立流程的原因 | 来源依据 |
|------|--------------|----------------------|----------|
| 投诉/客户关系工单 | 作为工单分类或标签，可备注是否涉及 CR（Customer Relationship，客户关系）。 | 投诉闭环、客户关系跟进、升级规则资料不足，独立做会扩大角色和状态。 | M1；需客户确认 |
| 政府/合约类工单 | 作为客户类型、项目类型、编号前缀或附件来源字段。 | 政府项目有特殊流程，但细则不足，第一版不应单独设计完整流程。 | M1、P；需客户确认 |
| 报价/采购/安装关联工单 | 仅保留报价编号、PO、ML、安装记录等关联字段或备注。 | 完整报价、采购、仓库、安装验收链路较长，属于二期或独立模块。 | M1、M3、P |
| 报销/考勤关联工单 | 仅可在处理记录或备注中关联费用/考勤信息，不形成审批流程。 | 报销、考勤、HR、Payroll（薪酬）属于另一条管理链路。 | M2、M6、M7 |
| 客户自助查询 | MVP 不做独立客户工作台，可作为二期候选。 | 当前主目标是内部工单闭环，客户入口、身份校验、展示范围未确认。 | M1、M4；需客户确认 |

## 3. MVP 角色收敛建议

### 3.1 MVP 角色

MVP 角色建议收敛为 4 类：

| MVP 角色 | 核心职责 | 核心操作 | 不建议在 MVP 承担的内容 |
|----------|----------|----------|--------------------------|
| CS / 工单受理人 | 接收客户请求，创建来电/服务记录，补充基础信息，判断是否进入派工。 | 新建工单、编辑受理信息、提交受理、标记取消或现场解决、查看相关工单。 | 不负责最终系统配置、复杂统计、全局权限管理。 |
| Team Leader / 分派与确认人 | 根据区域、团队、技能和任务情况分派师傅，跟进拒单和未完成事项，确认处理结果。 | 分派、重新分派、查看团队工单、确认处理完成、退回继续处理。 | 不承担完整管理员职责，不处理采购、财务、HR 流程。 |
| 师傅 / 处理人 | 执行现场服务、维修或特殊任务，反馈处理过程和结果。 | 接单、拒单并填写原因、填写处理记录、上传照片/附件、提交完成。 | 不负责派工、关闭工单、查看全局统计、修改客户主数据。 |
| 主管 / 管理员 | 查看整体进度和统计，处理异常，维护基础权限和配置。 | 查看全部或授权范围工单、查看基础统计、确认/关闭工单、管理基础角色权限。 | 不跳过流程随意修改处理记录；具体管理边界需客户确认。 |

### 3.2 暂不作为 MVP 独立工作台的角色

以下角色可以在 MVP 中作为字段、备注、关联人或二期角色出现，但不建议制作独立工作台：

| 角色 | MVP 处理方式 | 原因 |
|------|--------------|------|
| 客户 / 提交人 | 作为客户、联系人、来电人、客户公司等字段。 | 客户自助提交/查询涉及入口、身份校验、展示范围，需客户确认。 |
| CR / 客户关系 | 作为投诉/客户关系标签或跟进人字段。 | 投诉处理规则不足，不建议第一版单独设计工作台。 |
| 采购 / 仓库 | 作为 PO、ML、到货、物料备注等关联字段。 | 完整采购/仓库流程属于二期。 |
| 财务 | 作为报价、发票、报销等关联字段。 | 完整财务、报销、收付款不进入工单 MVP。 |
| HR | 不进入工单 MVP；如出现仅作为后续二期规划。 | HR、考勤、薪酬、绩效属于另一项目范围。 |

## 4. MVP 状态机收敛建议

### 4.1 MVP 主状态

MVP 状态机建议收敛为 7 个主状态：

| 状态 | 状态含义 | 可进入条件 | 可操作角色 | 下一状态 | 通知建议 |
|------|----------|------------|------------|----------|----------|
| 待受理 | CS 已记录客户请求，等待判断是否有效或是否需要派工。 | CS 创建来电/服务记录并提交。 | CS | 待分派、已取消、已关闭 | 通知 CS 或受理队列；渠道待确认。 |
| 待分派 | 工单确认需要外勤或后续处理，但尚未指定师傅。 | CS 受理后选择需要派工；或师傅拒单后退回。 | CS、Team Leader | 待接单、已取消 | 通知 Team Leader。 |
| 待接单 | 已分派给师傅，等待师傅接单或拒单。 | Team Leader 完成分派。 | 师傅、Team Leader | 处理中、待分派 | 通知师傅；拒单通知 Team Leader。 |
| 处理中 | 师傅已接单并正在处理，或处理过程仍需继续跟进。 | 师傅接单并开始处理。 | 师傅、Team Leader | 待确认、待分派、已取消 | 异常或长时间无更新可通知 Team Leader；超时规则待确认。 |
| 待确认 | 师傅提交处理结果和附件，等待 Team Leader 或主管确认。 | 师傅提交完成记录。 | Team Leader、主管 | 已关闭、处理中 | 通知确认人。 |
| 已关闭 | 工单完成并确认，流程结束，进入统计。 | Team Leader 或主管确认完成并关闭。 | Team Leader、主管 | 无；是否允许重开需确认 | 可通知相关内部角色；外部客户通知待确认。 |
| 已取消 | 工单无效、重复、客户取消或无需处理。 | CS 受理阶段或处理中确认无需继续。 | CS、Team Leader、主管 | 无；是否允许恢复需确认 | 通知相关内部角色。 |

### 4.2 状态收敛说明

- “师傅拒单”不作为独立主状态，而是动作。师傅拒单时必须填写拒单原因，工单状态回到“待分派”，由 Team Leader 重新分派。
- “客户补充”不作为 MVP 主状态。若需要客户补充资料，可先在“待受理”或“处理中”的处理记录中标记，是否设为独立状态待客户确认。
- “客户确认”不作为 MVP 主状态。MVP 默认由 Team Leader 或主管确认关闭；如客户明确要求外部确认闭环，再升级为独立状态或二期功能。
- “已驳回”不作为 MVP 主状态。无效、重复、不做、客户取消统一归入“已取消”，具体原因用字段记录。
- “草稿”不作为 MVP 主状态。CS 页面可允许未提交前暂存，但不进入正式状态机；是否需要草稿能力待客户确认。

## 5. 客户确认问题清单

### 5.1 必须确认后才能做原型的问题

| 优先级 | 问题 | 为什么必须先确认 |
|--------|------|------------------|
| 1 | MVP 是否确认只做内部工单闭环，不做客户自助提交/查询？ | 直接影响是否需要客户端页面、权限和展示范围。 |
| 2 | MVP 工单类型是否确认只保留“来电/服务记录、PM/维修/保养、CM/特殊任务”三类？ | 直接影响工单表单、列表筛选、流程图和统计口径。 |
| 3 | PM、CM 的业务定义和中文名称分别是什么？ | 影响客户是否能看懂类型，也影响师傅和 Team Leader 派工判断。 |
| 4 | 工单主流程是否确认为：CS 建单/来电记录 → CS 受理 → Team Leader 分派 → 师傅接单/拒单 → 师傅处理并上传记录 → Team Leader 或主管确认 → 关闭？ | 这是后续流程图和原型页面的主骨架。 |
| 5 | 分派规则以区域、团队、技能、空闲情况中的哪些为准？是否需要系统推荐 Team Leader？ | 影响分派页字段、按钮和确认逻辑。 |
| 6 | 工单完成后由谁确认关闭：Team Leader、主管、CS，还是必须客户确认？ | 影响状态机、关闭按钮权限和待确认页面。 |
| 7 | 师傅是否必须通过手机完成接单、拒单、上传照片和提交完成？ | 影响 HTML 原型是否必须做移动端优先页面。 |
| 8 | 哪些附件/照片是 MVP 必填，哪些客户或场所允许不拍照？ | 影响师傅处理页必填规则和验收标准。 |
| 9 | MVP 通知渠道使用什么：钉钉、电话、WhatsApp、邮件，还是仅在系统内提醒？ | 影响流程图通知节点和原型中的提示方式。 |
| 10 | 是否需要超时提醒进入 MVP？若需要，多久未接单、多久未处理算超时？ | 影响状态统计、提醒规则和基础看板。 |

### 5.2 可以边做原型边确认的问题

| 优先级 | 问题 | 可边做边确认的原因 |
|--------|------|--------------------|
| 11 | 工单编号规则是否需要日期、区域、客户或类型前缀？ | 原型可先用示例编号展示，最终规则后续确认。 |
| 12 | 工单列表筛选条件是否包含客户、区域、状态、类型、处理人、时间？ | 原型可先覆盖常用筛选，细节可在客户评审时调整。 |
| 13 | 未完成原因是否需要固定选项？例如材料不足、客户未签、等待报价、现场限制。 | 原型可先展示少量示例，最终枚举待客户补充。 |
| 14 | 处理记录是否需要固定模板，还是允许自由填写加附件？ | 原型可先采用“处理说明 + 附件/照片 + 结果”基础结构。 |
| 15 | 工单关闭后是否允许重开或补充记录？ | 不影响 MVP 主闭环，可在原型评审中确认操作边界。 |

### 5.3 二期规划问题

| 问题 | 建议归属 | 说明 |
|------|----------|------|
| 是否做客户自助查询、扫码查进度或客户确认关闭？ | 二期候选 | 需要客户身份、展示范围、外部通知等设计。 |
| 投诉/客户关系是否独立流程？ | 二期候选 | 可先用标签记录，待客户确认 CR 角色和投诉升级规则后再独立。 |
| 政府/合约项目是否需要独立编号、流程和报表？ | 二期候选 | 可先作为分类和附件来源，特殊流程后续细化。 |
| 报价、采购、PO、ML、到货、安装验收是否纳入同一系统？ | 二期或独立模块 | 链路长、角色多，不应压进工单 MVP。 |
| 报销 OCR、考勤、薪酬、绩效是否与工单联动？ | 二期或独立项目 | 资料支持存在讨论，但不属于工单第一版主流程。 |
| 自动客服、AI 图片检测、地址模糊查询、多系统深度集成是否建设？ | 远期规划 | 不影响当前 MVP 闭环，需另行评估投入和优先级。 |

| 章节 | 建议修改 | 修改原因 |
|------|----------|----------|
| 3. 建设目标 | 将目标聚焦为内部工单闭环，客户自助、报价采购、报销考勤仅作为二期或关联说明。 | 避免目标过宽，保持 MVP 可确认、可落地。 |

| 9. 页面清单 | 页面优先覆盖 CS 建单/受理、Team Leader 分派/确认、师傅处理、主管统计；客户端、采购/财务/HR 页面不进入 MVP 原型。 | 避免 HTML 原型做成多个系统入口。 |
| 10. 字段清单 | 字段按“主流程必填、可选、标签/关联、二期”重新分类。 | 当前部分字段如客户确认结果、PO、ML、费用信息容易被误认为 MVP 必做。 |
| 11. 权限矩阵 | 权限矩阵改为 4 个 MVP 角色版本；其他角色放到二期或备注。 | 原矩阵角色太多，客户确认成本高。 |
| 12. 通知机制 | MVP 只保留提交/分派/拒单/完成/确认/关闭等内部通知节点；客户外部通知待确认或二期。 | 通知渠道未明确，先围绕状态变化设计。 |
| 13. 报表统计需求 | MVP 统计保留总数、待受理/待分派/待接单/处理中/待确认/已关闭、按类型、按处理人、按区域、未完成原因。 | 统计必须服务主流程，不做复杂经营报表。 |
| 14. MVP 范围 | 用本澄清文档的主流程、3 类工单、4 类角色、7 个状态替换原 MVP 范围。 | 形成可直接进入流程图的边界。 |
| 15. 二期范围 | 将客户自助、投诉独立流程、政府合约独立流程、报价采购安装、报销考勤、HR/薪酬/绩效、AI/自动客服明确列为二期。 | 防止客户误以为第一版包含这些模块。 |
| 17. 需求追溯表 | 更新 RQ 编号，将 MVP 需求和二期候选分开；每项标明资料支持或待客户确认。 | 保持后续开发任务和客户确认有据可查。 |
| 18. 待客户确认问题 | 按“原型前必须确认、可边做边确认、二期规划”重新排序。 | 让客户评审更高效，避免被低优先级问题打断。 |
| 19. 验收标准 | 增加“已完成 MVP 收敛、原型仅覆盖主流程、二期不进入第一版”的验收条件。 | 确保需求规格进入下一阶段前满足宪法要求。 |

## 7. 下一步建议

建议下一步先执行：更新 `docs/01-requirements-spec.md`。

更新重点：

1. 将 MVP 范围改为 1 条主流程、3 类工单、4 类角色、7 个状态。
2. 将投诉/客户关系、政府/合约、报价/采购/安装、报销/考勤、客户自助查询降级为标签、关联字段或二期候选。
3. 将待客户确认问题按优先级重新排列，优先解决会影响原型结构的问题。

完成需求规格更新后，再生成 `docs/02-ticket-flow-analysis.md`，用于输出：

- 主流程 Mermaid 流程图。
- 角色泳道图。
- 状态流转图。
- 通知节点说明。
- 异常处理说明。
- 每个流程节点的验收标准。

暂不建议直接进入 HTML 原型。原因是当前仍有多个原型结构级问题需要客户确认，包括工单入口、PM/CM 定义、分派规则、关闭确认人、通知渠道和师傅移动端范围。若跳过需求收敛直接做原型，容易形成页面返工。
