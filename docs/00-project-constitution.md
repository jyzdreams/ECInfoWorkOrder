# EC 工单系统项目宪法

## 1. 项目背景

EC InfoTech Limited 是面向香港客户提供特低电压解决方案、安装及维修保养服务的企业，官网资料显示其业务覆盖保安系统、停车场系统、会所管理系统、电讯及广播服务、闭路电视、车牌识别、物联网及人工智能系统等领域。客户现有业务涉及大量现场服务、维修保养、安装、报价、采购、服务报告、完工记录和跨团队协作，因此需要更清晰的工单系统来承接日常服务流程。

根据 `materials` 文件夹中的客户沟通听记和纸质材料，客户当前存在纸质表单、电话/WhatsApp 记录、区域派工、师傅现场处理、拍照留痕、服务报告、报价/采购/结算关联、未完成事项跟踪等业务场景。客户也多次强调系统不能复杂到一线人员不愿使用，且过往供应商交付经历使客户对范围、报价、落地速度和验收边界非常敏感。

当前阶段不是开发阶段。本阶段目标是完成需求前置工作：项目原则、需求边界、流程原则、交付规范，为后续需求文档、可研分析、业务流程图和 HTML（超文本标记语言，用于网页原型）原型提供统一准则。

## 2. 项目目标

本项目要解决的核心问题是把 EC 现有服务流程从分散的电话、WhatsApp、纸质表单、Excel 和人工跟进，梳理为可确认、可追溯、可分阶段落地的工单业务闭环。

- 工单提交：资料明确客户问题来源包括电话、WhatsApp、客户现场沟通、政府或特殊项目渠道；是否允许客户直接自助提交属于合理推断，需客户确认。
- 工单受理：资料明确 CS（客服）负责记录来电、判断简单问题是否现场解决、发起需派工事项。
- 工单分派：资料明确按香港、九龙、新界等区域以及团队负责人进行派工，也存在 PM（Preventive Maintenance，预防性维护/常规维修）/CM（Corrective Maintenance 或特殊任务，具体客户口径需确认）、政府项目、跨区域或特殊任务等人工判断。
- 工单处理：资料明确师傅/外勤人员需要接单或拒单、说明原因、现场处理、上传照片或文件、填写处理记录。
- 工单反馈：资料明确需要通过处理结果、照片留痕、服务报告和未完成原因，让负责人和管理层看到工单卡在哪一步；是否向外部客户自动反馈状态属于合理推断，需客户确认。
- 工单关闭：资料明确完成后需要负责人或相关角色确认，最终形成流程闭环；关闭前是否需要客户签名或客户确认需客户确认。
- 工单统计：资料明确管理层关注未完成事项、状态、区域、负责人、预计完成时间等基础统计；更复杂的成本、绩效、自动分析属于二期候选。

本阶段的目标不是做“大而全”的 ITSM（IT Service Management，信息技术服务管理）平台，而是先建立 EC 工单业务的 MVP（Minimum Viable Product，最小可行版本）原则和确认路径。

## 3. 项目边界

第一阶段只做以下内容：

- 需求梳理：从客户沟通听记、纸质材料和客户官网中提炼已确认需求、合理推断和待确认问题。
- 可研评估：判断功能更适合钉钉配置、宜搭低代码、定制开发、人工流程保留，还是进入二期。
- 业务流程图：围绕工单建单、受理、分派、处理、反馈、关闭、查询、统计形成客户可读流程。
- HTML 原型：用于客户确认流程、字段、状态、角色和操作，不作为生产级前端。
- 后续开发任务拆分准备：为后续规格、计划、任务清单提供边界和验收依据。

当前明确不做真实开发，不创建真实接口，不设计数据库实现，不构建生产 UI（User Interface，用户界面），不进行系统上线或数据迁移。

## 4. 核心角色原则

以下角色来自客户沟通资料和现有纸质材料。后续需求文档必须继续验证每个角色的名称、权限和责任边界。

| 角色 | 核心职责 | 来源判断 |
|------|----------|----------|
| 客户 / 提交人 | 通过电话、WhatsApp、现场沟通或项目渠道提出服务、维修、安装、投诉或查询需求 | 来自客户沟通听记；直接自助提交属于合理推断，需客户确认 |
| CS / 客服 / 工单受理人 | 记录来电或客户问题，判断是否现场解决，创建工单，补充客户、区域、类型、附件等信息 | 来自客户沟通听记 |
| CR / 客户关系或投诉处理人 | 接收投诉类事项，协调复杂客户关系问题和后续跟进 | 来自客户沟通听记 |
| Team Leader / 团队负责人 | 按区域、团队、技能和任务类型接收或调整派工，跟进师傅执行情况 | 来自客户沟通听记 |
| 技术 / 师傅 / 处理人 | 接单或拒单，说明拒单原因，现场处理，上传照片或附件，填写处理结果 | 来自客户沟通听记和纸质样张 |
| 主管 / 管理员 | 查看未完成事项、状态统计、区域负载和异常工单，推动流程闭环 | 来自客户沟通听记 |
| 采购 / 仓库 | 与 PO（Purchase Order，采购单）、ML（客户现有资料中的关联编号，具体含义待客户确认）、到货、库存人工判断相关 | 来自客户沟通听记和纸质样张 |
| 财务 | 与报价、发票、报销、收款、付款、月度汇总相关 | 来自客户沟通听记和纸质样张 |
| HR / 人力资源 | 与考勤、员工资料、后续薪酬和绩效相关 | 来自客户沟通听记；不进入工单 MVP |

## 5. 需求优先原则

需求文档是后续流程图、HTML 原型、可研分析和开发任务拆分的源头。后续工作必须先确认业务需求，再画流程图，再做原型，最后才拆开发任务。

不得先做页面，再反推需求。不得因为某个平台或工具能实现某功能，就默认该功能属于客户需求。任何页面、按钮、字段、状态、通知和报表都必须能回到业务场景和客户资料。

需求表达必须优先使用业务语言，例如“来电记录”“派给九龙区负责人”“师傅拒单需说明原因”“完成后上传照片”，而不是优先使用抽象技术语言。

## 6. 需求追溯原则

后续每个核心需求必须标注来源，并至少归入以下三类之一：

- 来自客户沟通听记：例如来电记录、WhatsApp、纸质表单电子化、区域派工、PM/CM 分类、师傅接单/拒单、拍照留痕、未完成事项报表。
- 来自官网资料：仅用于客户行业背景、服务类型、业务性质和客户场景理解，不得单独扩展为系统功能。
- 合理推断，需客户确认：例如客户自助提交、外部客户状态通知、客户签名确认、具体通知渠道、具体权限矩阵、状态名称最终口径。

没有资料支撑的内容不得写成已确认需求。资料中没有明确提到的功能，只能写为“合理推断，需客户确认”或“二期候选，需客户确认”。

## 7. MVP 优先原则

第一版只做最小闭环，目标是让客户能确认工单主流程是否正确，而不是一次性建设完整 ITSM 平台。

MVP 范围包括：

- 建单：CS 或指定角色创建来电/服务/维修记录。
- 受理：判断是否需要派工，补充客户、区域、类型、问题描述、附件等基础信息。
- 分派：按区域、团队或人工判断分派给 Team Leader 或师傅。
- 处理：师傅接单、拒单、说明原因、现场处理、上传照片或处理记录。
- 反馈：向内部负责人反馈处理结果、异常原因和完成证据；外部客户自动反馈属于合理推断，需客户确认。
- 关闭：负责人确认完成并关闭工单；是否需要客户确认关闭需客户确认。
- 查询：按编号、客户、区域、状态、负责人、时间等查询工单。
- 基础统计：统计未完成工单、状态分布、区域分布、负责人和预计完成时间。

复杂能力放入二期，例如薪酬绩效、完整仓库库存、客户自助门户、自动客服、语音转写、WhatsApp 自动抓图、实时定位监控、ERP（Enterprise Resource Planning，企业资源计划）/CRM（Customer Relationship Management，客户关系管理）深度集成、AI（Artificial Intelligence，人工智能）自动生成工单或报告。

## 8. 流程设计原则

后续每条流程必须明确以下要素，缺一不可：

- 适用场景：该流程用于来电、投诉、维修、安装、政府项目、采购关联、报销关联还是其他场景。
- 发起角色：谁启动流程，例如客户、CS、CR、Team Leader、师傅、采购、财务。
- 参与角色：流程中会接收、处理、审批、确认或查看的人。
- 触发条件：什么事件使流程开始，例如客户来电、收到 WhatsApp、需要维修、师傅拒单、上传完成照片。
- 状态变化：从创建到关闭经过哪些状态，例如已创建、待受理、待分派、已分派、已接受、已拒绝、处理中、待确认、已完成、已关闭。
- 通知节点：哪些状态变化需要通知谁，以及通知内容是什么。
- 异常处理：拒单、资料不全、不能拍照、客户未确认、重复工单、无法完成、跨区域处理等情况如何处理。
- 结束条件：什么条件代表流程结束，例如负责人确认完成、客户确认、工单取消或转入二期流程。
- 验收标准：客户如何判断该流程正确，例如角色对、字段够、状态能流转、通知能触达、异常能处理。

流程图必须用中文节点表达，必要时可保留客户常用英文缩写，例如 CS、CR、PM、CM、PO。

## 9. 状态与权限原则

工单状态、角色权限和页面操作按钮必须一致。不能出现页面能操作但流程不允许的情况，也不能出现流程要求操作但页面没有入口的情况。

后续设计必须至少校验：

- 每个状态下谁可以查看、编辑、分派、接单、拒单、提交完成、退回、关闭。
- 每个按钮必须对应一个合法状态变化。
- 每个状态变化必须记录操作者、时间、原因或备注。
- 拒单、取消、退回、无法完成等异常状态必须有原因。
- 权限设计必须服务真实业务，不得为追求复杂而引入无资料支撑的多层审批。

具体权限矩阵属于合理推断，需客户确认。

## 10. 页面原型原则

HTML 原型必须服务于业务确认，而不是只追求好看。

原型必须做到：

- 页面体现真实业务流程，例如建单、受理、分派、处理、反馈、关闭、查询和统计。
- 页面字段来自客户资料、纸质表单或明确业务推断。
- 不追求复杂视觉设计，不做营销式页面，不用装饰掩盖流程不清。
- 不做无依据的大量功能，不把二期功能塞进 MVP 主流程。
- 菜单、按钮、字段、状态、提示语默认使用中文。
- 客户能通过原型确认“谁在什么情况下做什么，下一步流向哪里”。
- 开发能通过原型理解页面、字段、状态和操作边界。

如果某个页面只是为了展示好看，但不能帮助确认流程或字段，则不得进入当前阶段原型。

## 11. 通知与协作原则

通知机制必须围绕工单状态变化设计，而不是单独设计成泛消息系统。

后续至少需要识别以下通知节点：

- 工单创建后通知受理或分派角色。
- 工单分派后通知 Team Leader 或师傅。
- 师傅接受或拒绝后通知分派人或负责人。
- 工单处理中出现异常时通知负责人。
- 师傅提交完成后通知确认人。
- 工单关闭后通知相关内部角色。
- 未完成事项超期或长时间停留在某状态时是否提醒，属于合理推断，需客户确认。

通知渠道资料尚未完全明确。钉钉群、内部消息、电话、WhatsApp、邮件等渠道如何使用，需客户确认。当前只能确定通知必须跟随状态变化和责任人变化。

## 12. 验收原则

后续需求、流程图和 HTML 原型必须满足以下验收条件：

- 客户能看懂：使用业务语言，避免不必要技术术语。
- 开发能拆任务：每个功能能拆成明确页面、字段、状态、操作、规则和验收点。
- 流程能闭环：每条流程都有开始、处理、异常、结束和关闭。
- 角色能对应：每个动作都有明确角色负责。
- 状态能流转：每个状态都有进入条件、可操作角色和下一状态。
- MVP 范围清晰：第一版做什么、二期做什么、不做什么必须清楚。
- 资料能追溯：核心需求能标明来自客户沟通听记、官网背景或合理推断。

未满足上述条件的内容不得进入开发任务拆分。

## 13. 不做范围

第一阶段明确不做：

- 真实后端开发。
- 数据库实现。
- 真实接口设计或接口开发。
- 生产级前端或 UI 系统。
- 复杂 ITSM 平台。
- 资产管理。
- 知识库。
- SLA（Service Level Agreement，服务级别协议）计费。
- 多系统深度集成。
- 完整 HR、薪酬、绩效系统。
- 完整仓库库存系统。
- 客户自助门户。
- 自动客服。
- 语音转写自动建单。
- WhatsApp 截图自动抓取。
- 师傅实时定位监控。
- 未经客户确认的 AI 自动化。

除非客户资料明确要求并确认进入当前阶段，否则以上内容不得进入 MVP 范围。

## 14. 二期规划原则

二期功能必须基于客户确认后的优先级进入，不得在 MVP 中强行实现。

二期候选功能需要满足以下条件：

- 有明确客户来源或客户确认。
- 有清晰业务价值。
- 不影响 MVP 主流程落地。
- 有前置数据、人员习惯、预算、系统集成或合规条件说明。
- 能单独形成需求、流程、原型和验收标准。

当前可作为二期候选的方向包括：报销 OCR（Optical Character Recognition，光学字符识别）、考勤/薪酬/绩效、完整采购/仓库、客户状态查询、二维码查询、自动客服、AI 图片质量检查、AI 地址检索、ERP/CRM 深度集成等。上述方向仍需客户确认优先级和范围。

## 15. 语言与交付规范

- 所有正式交付文档默认使用简体中文。
- Mermaid（文本化流程图语法）流程图节点使用中文，必要时可保留英文缩写。
- HTML 原型页面文案使用中文，包括菜单、按钮、字段、状态、提示语。
- 技术术语首次出现需要解释中文含义，例如 HTML、MVP、SLA、OCR、ERP、CRM。
- 面向客户确认的内容优先使用业务语言。
- 面向开发执行的内容可以使用必要技术术语，但必须说明业务含义。
- 不得因为 spec-kit 默认模板为英文，就输出英文交付物。
- 除非用户明确要求英文版本，否则不输出英文交付物。

## 16. 治理规则

后续任何变更必须遵循以下顺序：

1. 先改需求文档，明确新增或变更需求的来源。
2. 再改流程图，确认角色、触发条件、状态流转、通知节点和验收标准。
3. 再改 HTML 原型，更新页面、字段、按钮、状态和提示语。
4. 最后才进入开发任务拆分。

任何新增功能必须标注来源或客户确认状态。没有来源的新增功能只能进入“合理推断，需客户确认”或“二期候选”，不得直接进入 MVP 或开发任务。

本项目所有正式输出必须接受以下治理检查：

- 是否符合需求优先原则。
- 是否能追溯到客户资料或明确标注待确认。
- 是否保持 MVP 范围清晰。
- 是否避免大而全 ITSM 化。
- 是否用中文表达并适合客户确认。
- 是否能支持后续流程图、原型和任务拆分。
