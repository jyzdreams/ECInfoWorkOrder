# EC 工单系统 MVP 可研分析文档

**版本**: 1.0.0  
**日期**: 2026-06-13  
**阶段**: MVP 可研分析与实施建议；不写代码、不生成 HTML 原型、不做数据库实现、不做接口开发。  
**输入依据**: `.specify/memory/constitution.md`、`docs/01-requirements-spec.md`、`docs/01-requirements-clarification.md`、`docs/01-requirements-analysis.md`、`docs/02-ticket-flow-analysis.md`、`materials` 客户资料。

## 1. 可研目标

本文件用于判断 EC 工单系统当前 MVP（Minimum Viable Product，最小可行版本）是否适合进入 HTML 原型和后续开发任务拆分。

本可研严格遵守当前 MVP 收敛范围：

- 1 条主流程：CS 建单 / 来电记录 -> CS 受理 -> Team Leader 分派 -> 师傅接单 / 拒单 -> 师傅处理并上传记录 -> Team Leader 或主管确认 -> 工单关闭 -> 基础统计。
- 3 类 MVP 工单：来电 / 服务记录工单、PM / 维修 / 保养类工单、CM / 特殊任务类工单。
- 4 类 MVP 角色：CS / 工单受理人、Team Leader / 分派与确认人、师傅 / 处理人、主管 / 管理员。
- 7 个 MVP 状态：待受理、待分派、待接单、处理中、待确认、已关闭、已取消。

本文件重点评估业务可行性、技术可行性、低代码 / 定制开发边界、数据与权限风险、实施排期和客户确认问题，为后续 HTML 原型页面结构说明和开发任务拆分提供依据。

## 2. 业务可行性分析

总体判断：当前 MVP 具备较高业务可行性。收敛后的主流程覆盖客户第一阶段最核心的诉求：把分散的客户请求转成可追踪工单，明确谁受理、谁分派、谁处理、谁确认、状态停在哪里，并能做基础统计。

| 业务环节 | 可行性 | 分析 | 建议 |
|----------|--------|------|------|
| CS 建单 / 来电记录 | 高 | 客户资料明确存在电话、WhatsApp、现场沟通、外部资料等入口，由 CS 手动记录最贴近现状，也能避免第一版接入外部系统。 | MVP 先以 CS 手动建单为唯一确定入口；客户自助提交放二期。 |
| CS 受理 | 高 | CS 判断是否现场解决、是否派工、是否取消，符合当前人工分流习惯。 | 原型中应突出“需要派工 / 现场解决 / 取消”三个受理动作。 |
| Team Leader 分派 | 高 | 资料显示当前存在按区域、团队、技能和人工判断分派，Team Leader 作为分派主责较合理。 | 需要客户确认 CS 是否也能直接分派，以及 Team Leader 的区域 / 团队范围。 |
| 师傅接单 / 拒单 | 高 | 师傅接单 / 拒单能让责任状态清晰，拒单原因能帮助 Team Leader 重新分派。 | 拒单必须填写原因；师傅端操作必须尽量简单。 |
| 处理记录和照片上传 | 高 | 纸质材料和沟通资料均体现现场照片、服务报告、完工记录的重要性。 | MVP 支持附件 / 图片上传，但哪些场景必填需客户确认。 |
| Team Leader 或主管确认关闭 | 中高 | 由内部负责人确认能保证闭环，避免客户确认规则不清导致流程卡住。 | MVP 默认 Team Leader 或主管确认；客户确认关闭不作为第一版必经节点。 |
| 基础统计 | 高 | 管理层需要查看工单数量、状态、处理人、区域等基础情况。 | 统计先服务管理可见性，不做成本、绩效、财务等复杂分析。 |

业务可行性结论：MVP 能覆盖第一阶段核心诉求，适合继续推进。主要前置确认点是 PM / CM 定义、分派权限、移动端优先级、附件 / 照片规则和通知渠道。

## 3. 技术可行性分析

| 能力 | 可行性 | 实现复杂度 | 风险说明 | 建议处理方式 |
|------|--------|------------|----------|--------------|
| 工单建单 | 高 | 低 | 字段较多时 CS 录入负担可能增加。 | MVP 表单只保留必填字段，其他作为可选或后续补充。 |
| 工单编号 | 高 | 低 | 编号规则未确认，政府 / 合约前缀不进 MVP。 | 原型先用自动编号示例；开发前确认最终规则。 |
| 状态流转 | 高 | 中 | 角色按钮与状态不一致会导致流程混乱。 | 严格按 7 个状态和角色权限设计按钮。 |
| 角色权限 | 高 | 中 | CS、Team Leader、主管之间可能存在权限重叠。 | 第一版只做 4 类角色；细按钮权限在客户确认后固化。 |
| 分派 | 高 | 中 | 自动推荐分派规则未明确，若做复杂算法会超范围。 | MVP 采用人工选择师傅；区域 / 技能作为筛选或参考字段。 |
| 接单 / 拒单 | 高 | 低 | 师傅端如果操作复杂，会影响使用。 | 手机端优先简化为“接单 / 拒单 + 原因”。 |
| 处理记录 | 高 | 中 | 固定模板未确认，强制字段过多会降低录入意愿。 | MVP 采用“处理说明 + 附件 / 图片 + 完成说明”基础结构。 |
| 图片 / 附件上传 | 高 | 中 | 文件大小、格式、不可拍照客户规则需确认。 | MVP 支持基础上传；限制和必填规则先做可配置或说明。 |
| 基础统计 | 高 | 低 | 统计口径若不统一，客户理解会不一致。 | 只按 7 状态、3 类型、处理人 / Team Leader、区域统计。 |
| 基础设置 | 中高 | 中 | 设置项过多会变成后台管理系统。 | MVP 仅保留类型、区域、角色、附件规则等简化设置。 |
| 移动端适配 | 中高 | 中 | 师傅是否必须手机操作仍需确认。 | 师傅操作页面建议移动端优先；管理页面可先桌面端。 |

技术可行性结论：MVP 技术难度整体中低，主要风险不在实现能力，而在流程口径、权限边界、通知渠道和一线录入复杂度。

## 4. 低代码 / 定制开发边界

| 功能 | 建议实现方式 | 原因 | MVP 是否必须 |
|------|--------------|------|---------------|
| 工单表单 | 低代码优先 | 表单字段、必填校验、附件上传适合低代码配置。 | 是 |
| 工单列表 | 低代码优先 | 列表、筛选、状态标签、基础查询属于常规能力。 | 是 |
| 工单详情 | 低代码 + 少量定制 | 详情页可低代码搭建，但状态动作、处理记录时间线可能需要定制。 | 是 |
| 状态流转 | 低代码 + 定制规则 | 7 状态清晰，低代码可配置；按钮权限和异常流转需规则控制。 | 是 |
| 权限控制 | 低代码 + 配置 | 4 类角色适合低代码权限模型，但细按钮权限需确认。 | 是 |
| 附件上传 | 低代码优先 | 多数平台支持基础附件。 | 是 |
| 图片上传 | 低代码优先，移动端需验证 | 图片上传本身常规，但手机拍照体验和压缩预览需验证。 | 是 |
| 分派逻辑 | 先人工处理 | MVP 不做自动派工；Team Leader 人工选择师傅即可。 | 是 |
| 通知提醒 | 先系统内提醒 | 渠道未确认，不宜先做钉钉 / WhatsApp / 邮件复杂通知。 | 是，限内部节点 |
| 统计看板 | 低代码优先 | 基础状态、类型、人员、区域统计适合低代码报表。 | 是 |
| 移动端师傅操作 | 低代码适配 + 必要定制 | 接单、拒单、拍照上传需保证手机可用，可能需要定制布局。 | 是，需客户确认优先级 |
| 基础设置 | 低代码优先 | 类型、区域、角色、附件规则适合配置表。 | 是，简化版 |
| 客户自助查询 | 二期 | 涉及外部权限、身份校验和展示范围。 | 否 |
| SLA 超时升级 | 二期 | 阈值、责任人、通知渠道未确认。 | 否 |

边界结论：MVP 可以优先采用低代码 / 配置化方式验证流程，复杂定制集中在状态按钮控制、移动端师傅体验、处理记录时间线和可能的通知对接。自动派工、外部客户门户、WhatsApp 自动通知、SLA 超时升级不进入第一版。

## 5. 数据可行性分析

| 数据类别 | 第一版是否必须 | 数据内容 | 客户需提供 | 校验建议 |
|----------|----------------|----------|------------|----------|
| 工单数据 | 必须 | 编号、类型、状态、客户、区域、描述、责任人、时间 | 需要确认编号规则和类型口径 | 编号、类型、状态、描述必填 |
| 客户数据 | 必须 | 客户公司、客户类型、地点 / 地址 | 需要提供现有客户清单或示例 | MVP 可先手工录入，不做强主数据校验 |
| 联系人数据 | 必须 | 联系人 / 来电人、联系方式 | 需要提供常见联系人格式 | 联系方式必填，格式先弱校验 |
| 区域数据 | 必须 | 香港、九龙、新界或更细区域 | 需要客户确认区域枚举 | 区域必填，枚举可配置 |
| 工单类型 | 必须 | 来电 / 服务记录、PM / 维修 / 保养、CM / 特殊任务 | 需要确认 PM / CM 定义 | 只能选择 3 类 MVP 类型 |
| 处理人 / 师傅数据 | 必须 | 师傅姓名、团队、区域、技能标签 | 需要客户提供师傅和团队资料 | MVP 可先只校验姓名和可用状态 |
| Team Leader 数据 | 必须 | Team Leader、负责区域 / 团队 | 需要客户提供团队关系 | MVP 需支持按区域或团队筛选 |
| 处理记录 | 必须 | 处理说明、完成说明、异常原因、记录时间 | 需要确认是否固定模板 | MVP 允许自由文本，关键节点必填 |
| 附件 / 图片 | 必须 | 图片、PDF、服务资料、现场照片 | 需要确认格式、大小、必填规则 | 第一版不做复杂质量校验 |
| 状态历史 | 必须 | 状态、操作人、操作时间、原因 | 不需要客户导入，系统生成 | 状态变化必须留痕 |
| 统计数据 | 必须 | 按状态、类型、处理人 / Team Leader、区域统计 | 不需要单独录入，由工单汇总 | 统计口径按 MVP 固定范围 |
| 投诉 / 政府 / 合约标签 | 可选 | 标签、备注、关联说明 | 需要确认是否展示 | 不作为工单类型强校验 |
| 报价 / PO / ML 关联字段 | 可选 / 二期 | 编号或备注 | 需要客户解释字段含义 | MVP 不做强校验 |

数据可行性结论：

- 第一版必须录入工单、客户、联系人、区域、类型、师傅、处理记录、附件 / 图片、状态历史。
- 客户主数据、师傅 / Team Leader 关系、区域枚举、PM / CM 定义需要客户提供或确认。
- 报价、采购、财务、HR 相关字段可以后续补充，不应在 MVP 中强校验。

## 6. 权限与角色可行性

4 类 MVP 角色足够支撑第一版闭环。

| 角色 | 核心权限 | 可能冲突 | 是否需要细分 | 可先简化项 | 必须客户确认 |
|------|----------|----------|--------------|------------|--------------|
| CS / 工单受理人 | 新建、受理、编辑受理信息、标记现场解决、标记取消、查看相关工单 | 是否可直接分派；是否可关闭现场解决工单 | 暂不细分 | CS 先不做复杂统计和设置 | 是否唯一建单入口、是否可分派、是否可关闭 |
| Team Leader / 分派与确认人 | 查看团队 / 区域工单、分派、重新分派、确认处理结果、查看团队统计 | 与主管在确认 / 关闭上的权限重叠 | 可按团队 / 区域配置范围 | 先不做复杂用户管理 | 是否一定负责分派和确认、负责范围 |
| 师傅 / 处理人 | 查看自己任务、接单、拒单、填写处理记录、上传附件、提交完成 | 是否允许编辑基础信息；是否允许取消 | 不需要细分 | 只保留自己任务视图 | 是否必须手机端操作、拒单是否必填原因 |
| 主管 / 管理员 | 查看全局、确认 / 关闭、查看统计、维护基础设置 | 与 Team Leader 的关闭权限重叠 | 可后续细分管理员 / 主管 | 先只做简化设置 | 是否有全局关闭和重开权限 |

权限可行性结论：

- MVP 不需要客户、CR、采购、财务、HR 独立工作台。
- 权限模型可先按 4 类角色实现。
- 需要客户确认 CS 与 Team Leader 的分派边界、Team Leader 与主管的关闭边界、已关闭后是否允许重开。

## 7. 通知机制可行性

MVP 内部通知节点清晰，但通知渠道未确认，因此不建议在第一版直接承诺多渠道自动通知。

| 通知节点 | 是否必须自动通知 | 可否先系统内提醒 | 渠道风险 | 建议 |
|----------|------------------|------------------|----------|------|
| 工单创建后通知 CS 或受理人 | 中 | 可以 | 如果 CS 自己创建，通知价值有限 | 原型表达为待受理队列或系统内提醒 |
| 工单进入待分派后通知 Team Leader | 高 | 可以 | 渠道未定 | MVP 必须有内部提醒，外部渠道待确认 |
| 工单分派后通知师傅 | 高 | 可以，但移动端体验重要 | 如果师傅不主动看系统，需要外部提醒 | MVP 先定义内部通知，钉钉 / WhatsApp 待确认 |
| 师傅接单 / 拒单后通知 Team Leader | 高 | 可以 | 拒单如果不及时通知会影响派工 | MVP 内部提醒必须保留 |
| 师傅提交处理结果后通知 Team Leader 或主管 | 高 | 可以 | 影响确认关闭效率 | MVP 内部提醒必须保留 |
| 工单关闭后进入统计 | 低 | 可以 | 不一定需要消息通知 | 作为状态变更和统计更新即可 |

通知可行性结论：

- MVP 必须保留通知节点，但不必一开始实现所有外部渠道。
- 可以先用系统内提醒、待处理队列、状态数量实现。
- 钉钉、WhatsApp、邮件等渠道需客户确认后再评估。
- 外部客户自动通知、WhatsApp 自动通知、SLA 超时升级不进入 MVP 确定功能，只能作为待确认或二期候选。

## 8. 移动端可行性

师傅场景具备明确移动端需求倾向，因为师傅需要在现场查看任务、接单 / 拒单、上传照片 / 附件、填写处理记录和提交完成。

| 师傅移动端能力 | 是否建议移动端优先 | 说明 |
|----------------|--------------------|------|
| 查看分派任务 | 是 | 师傅需要快速看到自己的待接单和处理中任务。 |
| 接单 / 拒单 | 是 | 操作必须简单，拒单原因可以选择 + 备注。 |
| 查看工单详情 | 是 | 需查看客户、地点、问题描述、附件和处理要求。 |
| 上传照片 / 附件 | 是 | 现场拍照上传是关键证据链。 |
| 填写处理记录 | 是 | 建议短文本 + 常用原因选项，减少录入负担。 |
| 提交完成 | 是 | 提交后进入待确认。 |
| 统计报表 | 否 | 师傅不需要统计看板。 |
| 基础设置 | 否 | 管理类页面可先桌面端。 |

页面适配建议：

- 必须适配手机：分派与接单视图中的师傅部分、工单详情 / 处理记录中的师傅操作、附件 / 图片上传、提交完成。
- 可以先桌面端：工单看板 / 首页、统计报表、基础设置。
- 建议同时适配桌面和手机：工单列表、工单详情。

HTML 原型建议：应提供至少一组师傅移动端关键页面效果，包括“待接单列表 / 工单详情 / 上传照片 / 提交完成”。管理端页面可以先以桌面端为主。

## 9. MVP 风险分析

| 风险 | 风险描述 | 影响 | 建议处理方式 | 是否阻塞原型 |
|------|----------|------|--------------|--------------|
| 客户流程口径未完全确认 | 分派人、确认人、关闭人仍有待确认项。 | 影响状态按钮和权限。 | 原型前确认 Top 10 问题。 | 部分阻塞 |
| PM / CM 定义不清 | PM、CM 的中文名称、边界和适用场景未最终确认。 | 影响工单类型、筛选和统计。 | 原型前确认定义，必要时用说明文字标注。 | 阻塞工单类型原型 |
| 通知渠道未定 | 钉钉、WhatsApp、邮件、系统内提醒未定。 | 影响技术方案和预算。 | MVP 先做系统内提醒，渠道列为待确认。 | 不阻塞流程原型 |
| 移动端优先级未定 | 师傅是否必须手机端完成全部操作未最终确认。 | 影响原型布局和开发工作量。 | 原型前确认；建议师傅端移动端优先。 | 部分阻塞 |
| 附件 / 照片必填规则未定 | 哪些工单必须照片、哪些场景不可拍照未明确。 | 影响表单校验和验收。 | 先支持上传，必填规则作为配置或待确认。 | 不阻塞原型 |
| 已关闭是否允许重开未定 | 关闭后是否可补充或重开未确认。 | 影响状态机和审计记录。 | MVP 可先不做重开，作为确认问题。 | 不阻塞初版原型 |
| Team Leader 权限范围未定 | 是否按区域 / 团队限制未明确。 | 影响权限设计和数据可见范围。 | 客户提供团队 / 区域关系。 | 部分阻塞 |
| 一线师傅是否愿意录入 | 界面复杂会降低录入意愿。 | 影响上线效果和数据完整性。 | 师傅端保持极简，只保留必要操作。 | 不阻塞，但影响设计 |
| 资料字段可能过多 | 字段太多会增加 CS 和师傅录入负担。 | 影响使用效率。 | 必填字段最小化，可选字段折叠。 | 不阻塞 |
| 二期功能反向挤入 MVP | 客户可能希望把报价、采购、客户自助等提前加入。 | 造成范围膨胀和延期。 | 坚持需求追溯和 MVP 边界，二期独立评估。 | 不阻塞，但需治理 |

## 10. MVP 开发复杂度评估

| 模块 | 复杂度 | 原因 | 是否 MVP 必须 | 是否建议第一版实现 |
|------|--------|------|----------------|--------------------|
| 工单表单 | 低 | 字段明确，适合配置化。 | 是 | 是 |
| 工单列表 | 低 | 常规列表、筛选、状态展示。 | 是 | 是 |
| 工单详情 | 中 | 需要基础信息、处理记录、附件、状态历史和角色操作。 | 是 | 是 |
| 状态流转 | 中 | 7 状态清晰，但需严格控制按钮和角色。 | 是 | 是 |
| 角色权限 | 中 | 4 角色可控，但权限边界需确认。 | 是 | 是 |
| 分派与接单 | 中 | 人工分派不复杂，移动端接单体验需打磨。 | 是 | 是 |
| 附件上传 | 中 | 基础上传可行，图片格式、大小、手机拍照体验需处理。 | 是 | 是 |
| 统计看板 | 低 | 只做基础统计，复杂报表不进 MVP。 | 是 | 是 |
| 基础设置 | 中 | 简化配置可行，避免扩成后台管理系统。 | 是 | 是，保持简化 |
| 移动端适配 | 中 | 师傅端必须简洁，上传照片和提交结果需适配。 | 是，需确认优先级 | 建议第一版实现关键师傅页面 |

复杂度结论：MVP 总体复杂度为中等偏低，适合进入原型和后续任务拆分。最大不确定性来自业务确认和移动端体验，而不是技术可实现性。

## 12. 4-6 周排期建议

以下排期为稳妥建议，不包含复杂外部系统集成、客户自助门户、WhatsApp 自动通知、SLA 超时升级等二期内容。

| 周期 | 工作内容 | 主要产出 | 客户配合 |
|------|----------|----------|----------|
| 第 1 周 | 需求与流程确认，冻结 MVP 边界。 | 客户确认版流程图、确认问题回复、基础资料清单。 | 确认主流程、3 类工单、4 角色、7 状态；提供区域、师傅、Team Leader 资料。 |
| 第 2 周 | HTML 原型页面结构设计与评审。 | 7 个页面原型结构、师傅移动端关键页面。 | 评审字段、按钮、状态、移动端操作。 |
| 第 3 周 | MVP 配置 / 开发第一轮。 | 工单表单、列表、详情、状态流转、基础权限。 | 提供测试账号和真实样例工单。 |
| 第 4 周 | 分派、接单 / 拒单、附件上传、基础统计完善。 | 可跑通主流程的 MVP 版本。 | 安排 CS、Team Leader、师傅试用。 |
| 第 5 周 | 测试、问题修正、权限和移动端体验调整。 | 测试记录、修正版本。 | 反馈问题、确认验收口径。 |
| 第 6 周 | 试用总结、上线准备或二期评估。 | MVP 验收清单、上线建议、二期候选清单。 | 确认是否进入上线或继续优化。 |

若客户能快速确认流程、权限和资料，项目可按 4-5 周推进；若通知渠道、移动端、权限边界反复调整，建议按 6 周预留。

## 14. 是否可以进入 HTML 原型

结论：可以有条件进入。

原因：

- 当前 MVP 已完成收敛，主流程、工单类型、角色、状态和页面范围清晰。
- `docs/02-ticket-flow-analysis.md` 已提供主流程图、三类工单流程、状态流转、异常流程和页面映射。
- 技术可行性较高，适合通过 HTML 原型进一步验证业务流程和一线操作。

进入 HTML 原型的条件：

1. 客户先确认 1 条主流程、3 类工单、4 类角色、7 个状态。
2. 客户确认 PM / CM 基本定义。
3. 客户确认师傅端是否移动端优先。
4. 客户确认附件 / 照片是否必填，以及是否存在不可拍照场景。
5. 客户确认通知渠道先按系统内提醒表达，还是需要在原型中体现钉钉等渠道。

如果以上问题暂未完全确认，也可以先做低保真 HTML 原型结构，但必须在原型中标注待确认项，避免客户误以为已确定。

## 15. 下一步建议

建议下一步优先生成：HTML 原型页面结构说明。

推荐顺序：

1. 先生成 HTML 原型页面结构说明，明确 7 个页面的字段、按钮、状态、角色可见性和跳转关系。
2. 再生成 `prototype/index.html` 及相关静态原型页面。
3. 同步准备客户确认版流程图，用于客户会议确认。
4. 客户确认原型后，再进入开发任务拆分。

暂不建议直接进入开发任务拆分。开发任务应建立在客户确认后的流程图和 HTML 原型之上，避免把待确认规则写成开发范围。
