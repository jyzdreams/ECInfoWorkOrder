# EC 工单系统 HTML 原型页面结构说明

**版本**: 1.0.0  
**日期**: 2026-06-13  
**阶段**: HTML 原型页面结构说明；不写实际 HTML 代码、不生成 `prototype/index.html`、不做数据库、接口或后端设计。  
**输入依据**: `.specify/memory/constitution.md`、`docs/01-requirements-spec.md`、`docs/02-ticket-flow-analysis.md`、`docs/03-feasibility.md`、`materials` 客户资料。

## 1. 原型目标

本原型用于帮助客户确认 EC 工单系统 MVP（Minimum Viable Product，最小可行版本）的业务流程、页面字段、状态流转、角色操作和第一版范围。

本原型不是生产级前端，不用于真实数据录入、真实通知、真实权限控制、数据库实现或接口联调。后续 `prototype/index.html` 必须以本文件为唯一页面结构依据，不得自行加入二期页面或无来源功能。

原型固定覆盖：

- 1 条主流程：CS 建单 / 来电记录 -> CS 受理 -> Team Leader 分派 -> 师傅接单 / 拒单 -> 师傅处理并上传记录 -> Team Leader 或主管确认 -> 工单关闭 -> 基础统计。
- 3 类 MVP 工单：来电 / 服务记录工单、PM / 维修 / 保养类工单、CM / 特殊任务类工单。
- 4 类 MVP 角色：CS / 工单受理人、Team Leader / 分派与确认人、师傅 / 处理人、主管 / 管理员。
- 7 个 MVP 状态：待受理、待分派、待接单、处理中、待确认、已关闭、已取消。
- 7 个原型页面：工单看板 / 首页、工单列表、新建工单 / 来电记录、工单详情 / 处理记录、分派与接单视图、统计报表、基础设置，简化版。

## 2. 原型设计原则

1. 以流程确认为主，页面必须能清楚表达谁在什么状态下做什么操作。
2. 页面要清晰、商务、克制，适合香港企业客户评审。
3. 不追求复杂视觉设计，不使用花哨动效，不做营销式展示页。
4. 不做二期功能主页面，不出现客户门户、CR 工作台、采购 / 财务 / HR 工作台。
5. 所有页面文案、菜单、按钮、字段、状态、提示语默认使用简体中文。
6. 师傅端操作要简单，优先突出待接单、接单、拒单、上传照片、提交结果。
7. 状态和按钮必须对应，不能出现当前状态不允许但页面上可操作的按钮。
8. 原型使用模拟数据，不连接真实后端。
9. 原型用于客户确认业务，不作为最终视觉规范或开发完成标准。

## 3. 原型整体布局

建议后续 `prototype/index.html` 使用单页 HTML 原型，通过左侧菜单或顶部 Tab 切换 7 个页面。

| 区域 | 结构说明 | 原型要求 |
|------|----------|----------|
| 左侧菜单 / 顶部导航 | 展示 7 个页面入口：工单看板、工单列表、新建工单、工单详情、分派与接单、统计报表、基础设置。 | 菜单文字使用中文；当前页面需要高亮。 |
| 角色切换模拟 | 提供 CS、Team Leader、师傅、主管 4 个角色切换。 | 切换角色后，按钮和可见页面应变化；不做真实登录。 |
| 主内容区 | 展示当前页面的业务内容。 | 页面以表格、表单、状态卡、时间线为主。 |
| 状态筛选 | 在看板、列表、统计中提供 7 状态筛选。 | 状态名称必须固定为：待受理、待分派、待接单、处理中、待确认、已关闭、已取消。 |
| 页面切换方式 | 单页内通过 Tab、侧边栏或导航按钮切换。 | 不要求多 HTML 文件。 |
| 模拟数据 | 内置 8-12 条工单数据。 | 必须覆盖 3 类工单、7 个状态、至少 3 个区域、3 个师傅、2 个 Team Leader。 |
| 响应式要求 | 桌面端优先覆盖管理页面，师傅端关键操作需要手机宽度效果。 | 原型至少体现师傅移动端的简洁操作区。 |

建议布局：

- 顶部：项目名称、当前模拟角色、当前日期、全局搜索。
- 左侧：页面菜单。
- 主区域：页面内容。
- 右侧或页面顶部：状态筛选、角色说明、待确认提示。
- 底部：仅可放置原型说明，不放开发技术信息。

## 4. 页面 1：工单看板 / 首页

| 项目 | 说明 |
|------|------|
| 使用角色 | 主管、Team Leader、CS |
| 页面目的 | 快速查看工单整体情况、当前待办和异常 / 待处理工单。 |
| 对应流程节点 | 基础统计、待处理入口、进入列表或详情。 |
| 是否 MVP 必须 | 是 |

核心模块：

| 模块 | 内容 | 说明 |
|------|------|------|
| 统计卡片 | 工单总数、待受理、待分派、待接单、处理中、待确认、已关闭、已取消。 | 每个卡片可点击进入对应状态的工单列表。 |
| 今日待办 | 待受理、待分派、待确认等需要当前角色处理的工单。 | 根据角色切换展示不同待办。 |
| 异常 / 待处理工单 | 拒单、确认不通过、信息不足、无法完成等示例。 | 异常不新增状态，用标签或原因说明。 |
| 按工单类型统计 | 来电 / 服务记录、PM / 维修 / 保养、CM / 特殊任务。 | 不出现投诉、政府、报价采购等独立类型。 |
| 按区域统计 | 至少展示香港、九龙、新界 3 个区域。 | 区域仅用于筛选和统计。 |

核心操作：

- 查看工单详情。
- 快速进入工单列表。
- 新建工单。
- 按状态筛选。
- 按角色查看待办。

页面注意事项：

- 师傅角色可以不进入完整看板，或只显示“我的待接单 / 处理中任务”简化视图。
- 不展示成本、绩效、财务、SLA（Service Level Agreement，服务级别协议）计费等二期统计。

| 项目 | 说明 |
|------|------|
| 使用角色 | CS、Team Leader、师傅、主管 |
| 页面目的 | 统一查看、筛选和进入工单详情。 |
| 对应流程节点 | 查询、全流程查看、进入详情操作。 |
| 是否 MVP 必须 | 是 |

核心筛选：

| 筛选项 | 说明 |
|--------|------|
| 状态 | 7 个 MVP 状态。 |

| 区域 | 香港、九龙、新界等模拟区域。 |
| 处理人 | 师傅 / 处理人。 |
| Team Leader | 分派与确认人。 |
| 时间范围 | 创建时间或更新时间。 |
| 关键词 | 工单编号、客户、联系人、问题描述。 |

表格字段：

| 字段 | 说明 |
|------|------|
| 工单编号 | 示例：WO-20260613-001。 |
| 工单类型 | 来电 / 服务记录、PM / 维修 / 保养、CM / 特殊任务。 |
| 客户 / 联系人 | 客户公司 + 联系人。 |
| 区域 | 香港、九龙、新界等。 |
| 状态 | 7 个状态之一。 |
| 当前负责人 | CS、Team Leader 或师傅。 |
| 创建时间 | 模拟日期时间。 |
| 更新时间 | 模拟日期时间。 |
| 操作 | 查看详情，以及根据角色和状态显示的按钮。 |

核心操作：

- 查看详情。
- 新建工单。
- 根据角色显示不同操作按钮。
- 通过状态卡或筛选条件缩小列表。

角色显示要求：

- CS：重点显示待受理、自己创建的工单。
- Team Leader：重点显示待分派、待确认、团队 / 区域工单。
- 师傅：重点显示待接单、处理中、分派给自己的工单。
- 主管：可查看全部模拟工单。

## 6. 页面 3：新建工单 / 来电记录

| 项目 | 说明 |
|------|------|
| 使用角色 | CS |
| 页面目的 | 记录客户来电、WhatsApp、现场沟通或服务请求，并进入受理判断。 |
| 对应流程节点 | CS 建单 / 来电记录、CS 受理。 |
| 是否 MVP 必须 | 是 |

字段分组：

| 分组 | 字段 | 说明 |
|------|------|------|
| 基本信息 | 工单编号、创建时间、CS / 受理人 | 编号和时间可自动生成模拟值。 |
| 客户与联系人 | 客户公司、联系人 / 来电人、联系方式 | 必填。 |
| 工单类型 | 来电 / 服务记录、PM / 维修 / 保养、CM / 特殊任务 | 必填，只允许 3 类。 |
| 区域与地址 | 区域、地点 / 地址 | 区域必填，地址可按原型需要展示。 |
| 问题描述 | 问题描述 | 必填。 |
| 附件 / 图片 | 上传附件、上传图片 | MVP 支持，但是否必填待客户确认。 |
| 是否需要派工 | 需要派工、现场解决、取消 | 用于受理判断。 |
| 备注 / 标签 | 备注、投诉标签、政府 / 合约标签 | 可选，不作为独立流程。 |

必填字段：

- 工单类型。
- 客户 / 联系人。
- 联系方式。
- 区域。
- 问题描述。

可选字段：

- 附件 / 图片。
- 备注。
- 标签。
- 地址补充说明。

核心操作：

| 按钮 | 状态变化 | 说明 |
|------|----------|------|
| 保存为待受理 | 新记录 -> 待受理 | 仅保存建单信息，等待后续受理。 |
| 创建并进入受理 | 新记录 -> 待受理，并展示受理操作区 | 推荐主按钮。 |
| 进入待分派 | 待受理 -> 待分派 | CS 判断需要派工后使用。 |
| 标记现场解决 | 待受理 -> 已关闭 | 现场或电话中已解决。 |
| 取消 | 待受理 -> 已取消 | 无效、重复、客户取消等。 |

页面注意事项：

- 不设计客户自助提交入口。
- 不设计 WhatsApp 自动抓图或语音转写。
- 标签只做辅助说明，不引出独立流程。

## 7. 页面 4：工单详情 / 处理记录

| 项目 | 说明 |
|------|------|
| 使用角色 | CS、Team Leader、师傅、主管 |
| 页面目的 | 查看工单全貌、状态流转、处理记录和附件，并执行当前状态允许的操作。 |
| 对应流程节点 | 全流程。 |
| 是否 MVP 必须 | 是 |

核心模块：

| 模块 | 内容 |
|------|------|
| 工单基础信息 | 工单编号、类型、客户、联系人、联系方式、区域、地址、问题描述。 |
| 当前状态 | 显示 7 个状态之一，并用状态颜色区分。 |
| 当前负责人 | CS、Team Leader、师傅或主管。 |
| 处理记录时间线 | 记录受理、分派、接单、拒单、处理、上传、确认、关闭等节点。 |
| 附件 / 图片 | 展示上传文件、图片缩略图、上传人、上传时间。 |
| 状态流转记录 | 显示每次状态变化、操作人、时间、原因。 |
| 备注 | 信息不足、无法完成、取消原因、拒单原因等。 |

按角色显示按钮：

| 角色 | 当前状态 | 可用按钮 | 操作后状态 |
|------|----------|----------|------------|
| CS | 待受理 | 进入待分派 | 待分派 |
| CS | 待受理 | 标记现场解决 | 已关闭 |
| CS | 待受理 | 取消 | 已取消 |
| CS | 待受理 / 处理中 | 补充信息 | 状态不变 |
| Team Leader | 待分派 | 分派 | 待接单 |
| Team Leader | 待接单 / 处理中 | 重新分派 | 待接单或待分派 |
| Team Leader | 待确认 | 确认关闭 | 已关闭 |
| Team Leader | 待确认 | 退回继续处理 | 处理中 |
| 师傅 | 待接单 | 接单 | 处理中 |
| 师傅 | 待接单 | 拒单 | 待分派 |
| 师傅 | 处理中 | 提交处理结果 | 待确认 |
| 主管 | 待确认 | 确认关闭 | 已关闭 |
| 主管 | 待受理 / 待分派 / 处理中 | 取消 | 已取消 |
| 主管 | 已关闭 / 已取消 | 查看 | 状态不变 |

按钮规则：

- 拒单必须填写拒单原因。
- 取消必须填写取消原因。
- 确认不通过必须填写退回原因。
- 提交处理结果建议填写完成说明并上传附件 / 图片。
- 已关闭和已取消原则上只读；是否允许重开待客户确认。

## 8. 页面 5：分派与接单视图

| 项目 | 说明 |
|------|------|
| 使用角色 | Team Leader、师傅 |
| 页面目的 | 让 Team Leader 快速分派任务，让师傅快速接单 / 拒单。 |
| 对应流程节点 | Team Leader 分派、师傅接单 / 拒单。 |
| 是否 MVP 必须 | 是 |

Team Leader 视图：

| 模块 | 内容 | 操作 |
|------|------|------|
| 待分派工单列表 | 工单编号、类型、客户、区域、问题摘要、创建时间。 | 选择工单。 |
| 区域 / 类型 / 技能参考 | 区域、工单类型、技能标签、特殊要求。 | 作为人工分派参考，不做自动推荐。 |
| 师傅选择 | 师傅姓名、所属团队、负责区域、当前任务数。 | 选择师傅。 |
| 分派备注 | 派工说明、注意事项。 | 可选填写。 |
| 分派按钮 | 将工单分派给师傅。 | 待分派 -> 待接单。 |

师傅视图：

| 模块 | 内容 | 操作 |
|------|------|------|
| 待接单任务 | 分派给当前师傅的工单。 | 查看简要信息。 |
| 任务详情简要 | 客户、地址、问题描述、附件、处理要求。 | 展开详情。 |
| 接单按钮 | 师傅确认接单。 | 待接单 -> 处理中。 |
| 拒单按钮 | 师傅拒绝任务。 | 待接单 -> 待分派。 |
| 拒单原因 | 原因选择 + 备注。 | 拒单必填。 |

核心状态变化：

- 待分派 -> 待接单：Team Leader 分派师傅。
- 待接单 -> 处理中：师傅接单。
- 待接单 -> 待分派：师傅拒单后回退，Team Leader 重新分派。

移动端要求：

- 师傅视图必须适合手机查看。
- 接单 / 拒单按钮要清晰，避免误操作。
- 拒单原因输入不能太复杂。

## 9. 页面 6：统计报表

| 项目 | 说明 |
|------|------|
| 使用角色 | 主管、Team Leader |
| 页面目的 | 查看 MVP 基础统计，支撑管理层了解工单整体情况。 |
| 对应流程节点 | 基础统计。 |
| 是否 MVP 必须 | 是 |

统计范围：

| 统计项 | 展示方式 |
|--------|----------|
| 按状态统计 | 7 个状态数量和占比。 |
| 按工单类型统计 | 3 类 MVP 工单数量。 |
| 按区域统计 | 香港、九龙、新界等区域数量。 |
| 按处理人统计 | 师傅处理数量、处理中数量、待确认数量。 |
| 按 Team Leader 统计 | Team Leader 负责工单数量和状态分布。 |

筛选条件：

- 时间范围。
- 工单类型。
- 区域。
- Team Leader。
- 处理人。

不做：

- 成本统计。
- 绩效统计。
- 财务统计。
- SLA 计费。
- 多维复杂分析。
- 客户自助查询报表。

页面注意事项：

- 统计图表只需清晰，不追求复杂动态图。
- 点击统计项可跳转到工单列表并带入筛选条件。

## 10. 页面 7：基础设置，简化版

| 项目 | 说明 |
|------|------|
| 使用角色 | 主管 / 管理员 |
| 页面目的 | 维护原型所需基础数据，不做复杂后台。 |
| 对应流程节点 | 支撑工单类型、区域、角色、附件规则等基础口径。 |
| 是否 MVP 必须 | 是，简化版 |

设置项：

| 设置项 | 内容 | 说明 |
|--------|------|------|
| 工单类型 | 来电 / 服务记录、PM / 维修 / 保养、CM / 特殊任务 | 固定 3 类，不新增二期类型。 |
| 区域 | 香港、九龙、新界等 | 用于分派、筛选和统计。 |
| 师傅 / 处理人 | 师傅姓名、区域、技能标签、可用状态 | 用于人工分派参考。 |
| Team Leader | 姓名、负责区域 / 团队 | 用于分派和统计。 |
| 状态说明 | 7 个 MVP 状态解释 | 只读说明即可。 |
| 通知规则说明 | MVP 内部通知节点说明 | 渠道待客户确认。 |
| 附件规则 | 是否允许拍照、附件是否必填 | 作为原型说明，具体规则待确认。 |

不做：

- 复杂权限系统。
- 多组织架构。
- API（Application Programming Interface，应用程序接口）集成设置。
- 财务 / 采购 / HR 设置。
- 客户门户设置。
- SLA 配置。
- 自动通知渠道配置。

| 待接单 | 师傅 | 接单 | 处理中 | 不需要 | 通知 Team Leader | 分派与接单视图、工单详情 |
| 待接单 | 师傅 | 拒单 | 待分派 | 必填拒单原因 | 通知 Team Leader | 分派与接单视图、工单详情 |
| 处理中 | 师傅 | 上传附件 / 图片 | 处理中 | 可选说明 | 不需要 | 工单详情 / 处理记录 |
| 处理中 | 师傅 | 提交处理结果 | 待确认 | 必填完成说明，附件规则待确认 | 通知 Team Leader 或主管 | 工单详情 / 处理记录 |
| 处理中 | Team Leader、主管 | 重新分派 | 待分派 | 必填原因 | 通知 Team Leader | 工单详情、分派与接单视图 |
| 处理中 | 主管 | 取消 | 已取消 | 必填取消原因 | 可系统内记录 | 工单详情 |
| 待确认 | Team Leader、主管 | 确认关闭 | 已关闭 | 可选确认说明 | 进入统计 | 工单详情 / 处理记录 |
| 待确认 | Team Leader、主管 | 退回继续处理 | 处理中 | 必填退回原因 | 通知师傅 | 工单详情 / 处理记录 |
| 已关闭 | CS、Team Leader、师傅、主管 | 查看 | 已关闭 | 不需要 | 不需要 | 工单详情、列表、统计报表 |
| 已取消 | CS、Team Leader、主管 | 查看 | 已取消 | 不需要 | 不需要 | 工单详情、列表、统计报表 |

说明：

- 已关闭后是否允许重开待客户确认，原型不默认提供重开按钮。
- 客户补充资料不作为状态，仅通过备注或处理记录体现。
- 外部客户自动通知不作为 MVP 确定功能。

## 14. 原型视觉风格建议

| 项目 | 建议 |
|------|------|
| 整体风格 | 商务后台风格，清晰、轻量、专业。 |
| 客户适配 | 适合香港企业客户展示，避免过度互联网化或营销化。 |
| 色彩 | 使用克制的中性色背景，状态色清晰但不过度鲜艳。 |
| 状态颜色 | 待受理、待分派、待接单使用提醒色；处理中使用蓝色；待确认使用橙色；已关闭使用绿色；已取消使用灰色。 |
| 表格 | 字段不溢出，重要字段优先展示，长文本可截断并展开。 |
| 表单 | 必填项清晰，可选项折叠或分组。 |
| 按钮 | 主按钮明确，危险操作如取消需二次确认或原因输入。 |
| 移动端师傅视图 | 简洁、按钮大、信息少而关键，适合现场快速操作。 |
| 动效 | 不使用花哨动效，仅保留必要的切换和反馈。 |

## 15. 不进入原型主流程的内容

以下内容不进入 MVP 原型主流程：

- 投诉独立工作台。
- 政府 / 合约独立流程。
- 报价 / 采购 / 安装完整流程。
- 报销 / 考勤流程。
- 客户自助门户。
- WhatsApp 自动通知。
- SLA 超时升级。
- AI 图片检测。
- ERP / CRM 深度集成。

这些内容最多作为标签、备注、关联字段或二期说明，不得设计成主页面、主菜单或主流程。

## 16. 客户确认点

原型前需要客户确认的问题：

1. 是否确认当前 7 个页面。
2. 是否确认 CS 是第一版主要建单入口。
3. 是否确认 Team Leader 负责分派和确认。
4. 是否确认师傅端要移动端优先。
5. 是否确认附件 / 照片必填规则。
6. 是否确认通知先用系统内提醒。
7. 是否确认已关闭是否允许重开。
8. 是否确认已取消是否进入统计。
9. 是否确认 PM / CM 定义。
10. 是否确认基础设置范围。

## 17. 下一步建议

完成本文件后，可以进入 `prototype/index.html` 静态 HTML 原型生成。

进入原型生成的建议条件：

1. 以本文件作为唯一页面结构依据。
2. 仅生成 7 个 MVP 页面。
3. 使用本文件定义的模拟数据。
4. 按状态与按钮映射表控制可见按钮。
5. 师傅端至少提供移动端关键操作效果。
6. 二期功能不得出现在主菜单或主流程中。

不建议在生成原型时新增页面、新增状态、新增工单类型或新增角色。如果客户在原型评审中提出新增功能，应先回到需求文档记录来源和确认状态，再判断是否进入二期。
