# 冲突点与待确认问题

## 冲突点/待确认点

| 编号 | 问题 | 材料表现 | 影响 | 建议 |
|---|---|---|---|---|
| Q1 | 主管是否必须确认完工 | M4建议简化，避免主管重复审批；M5设计“最终由负责人手动选择已完成”。 | 影响状态流转和师傅闭环速度。 | 做成可配置：试点阶段默认Leader确认，低风险工单可抽检。 |
| Q2 | PM/CM定义边界 | M6定义PM常规维修、CM特别任务，但具体场景和子类未完整列出。 | 影响表单选项、派单、报表。 | 与CS/CR/Leader做一次流程工作坊，按真实工单样例归类。 |
| Q3 | 五个团队分派规则 | M6提到默认五个团队规则，I19有区域联系人表，但未给完整映射。 | 影响自动推荐Leader。 | 先做配置表，由客户填区域码、Leader、成员、技能。 |
| Q4 | 普通工单编号与项目编号关系 | M6给出工单编号日期+序号，M4给出项目编号区域码+年份+序列号。 | 影响检索和财务/采购关联。 | 工单编号和项目编号分离，工单可关联项目。 |
| Q5 | 政府合约编号前缀 | M6示例为前缀字母+ECA+日期+序号，但“前缀字母”具体规则未列。 | 政府工单编号不完整。 | 客户提供政府合约/部门前缀表。 |
| Q6 | 报销是否一期全员电子化 | M6提到部分老师傅仍倾向手写；M5 demo可AI识别。 | 影响培训、流程强制性。 | 一期允许拍照上传手写单+人工填写；AI识别二期试点。 |
| Q7 | 是否采购宜搭/专业版 | M3提到宜搭二期才采购，M5又需要一期PM/CM执行流程；近800人超500上限。 | 影响实施可行性和账号覆盖。 | 商务上先确认钉钉专业版/宜搭许可和容量。 |
| Q8 | 图片保存期限与容量 | M1提及月均可能1TB，需本地归档下载；未给保留策略。 | 影响成本和性能。 | POC期间测真实图片量，制定在线保留+归档周期。 |
| Q9 | 不可拍照客户清单 | M5提到监狱等特殊场所，I26有惩教署报价，但清单未提供。 | 影响照片必填规则。 | 客户提供初始不可拍照客户/项目清单。 |
| Q10 | 客户自助查询一期是否要公开链接 | M1、M4提到QR/公开URL；M6确认自动化客服本期不做。 | 影响外部权限和安全。 | 一期不做外部客户自助，只预留字段。 |
| Q11 | 采购/仓库边界 | M6确认只做采购管理关联，不做库存；M4又提到到货全公司查询。 | 影响采购模块深度。 | 一期只记录PO、ML、预计/实际到货和状态。 |
| Q12 | 报价特殊流程 | M5提到有时未确认报价先施工；M4强调确认后不可直接修改需版本。 | 影响合规和版本控制。 | 一期只标记“先施工后报价/特殊情况”，二期做完整版本控制。 |
| Q13 | 账号离职数据保留 | M5提到当前保留离职账号，未来可能清理inactive。 | 影响历史工单、考勤、费用追溯。 | 历史记录保存人员快照，账号可禁用不删除业务数据。 |

|---|---|
| P0 | 试点范围：先选哪个区域/团队、多少师傅、多少CS/Leader。 |
| P0 | PM/CM工单样例各10条，用于确认字段和流转。 |
| P0 | 区域码与Team Leader/师傅映射表。 |
| P0 | 工单编号、政府前缀、项目编号、PO、ML、S/R编号最终规则。 |
| P0 | 客户/地址/屋苑基础清单，含繁体中文名、英文别名、行政区。 |
| P0 | 不可拍照客户/项目清单。 |
| P0 | 是否强制Team Leader确认完工。 |
| P0 | 钉钉版本、宜搭许可、通讯录人数、存储容量。 |
| P1 | 报销是否一期试点，试点人群和审批人。 |
| P1 | 采购/PO字段和采购负责人。 |
| P1 | 管理层看板最关注的5个指标。 |
| P2 | 客户自助查询、公开链接、QR Code的安全要求。 |
| P2 | AI/OCR预算、准确率要求和人工审核流程。 |
