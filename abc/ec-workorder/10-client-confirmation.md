# 客户确认范围文件

本文件用于客户确认EC InfoTech工单系统一期范围。未确认或新增需求进入变更池，不直接加入一期。

## 1. 项目目标

一期目标是建立工单闭环：CS手工建单、PM/CM分类、区域/Team Leader推荐、Leader派单、师傅移动端执行、拍照/不可拍照豁免、处理结果提交、状态追踪、查询和基础权限日志。

一期不以替代ERP、仓库、报价、财务、HR系统为目标。

## 2. 一期确认范围

### P0 必须上线

| 范围 | 说明 |
|---|---|
| CS来电/WhatsApp手工建单 | 手工录入，不做自动抓取。 |
| 工单自动编号 | 普通小区类和政府合约类规则。 |
| PM/CM分类 | 按客户确认后的定义执行。 |
| 区域/Team Leader推荐 | 根据地址/区域码推荐，可手动调整。 |
| Leader派单/改派 | 支持派给师傅、改派、拒单处理。 |
| 师傅移动端接单/拒绝 | 移动端极简操作。 |
| 到场记录 | 记录到场时间和位置文字/选点。 |
| 拍照上传/不可拍照豁免 | 可拍照客户上传照片，不可拍照客户填写原因。 |
| 处理结果提交 | 师傅填写结果和后续需要。 |
| 工单状态流转 | 支持主状态和处理中子状态。 |
| 工单列表查询 | 按时间、状态、区域、客户、负责人、关键词查询。 |
| 基础权限和操作日志 | 独立账号、按角色和团队授权。 |

### P1 一期可选

| 范围 | 确认方式 |
|---|---|
| 工单池主动接单 | 客户确认启用后加入。 |
| Outstanding原因看板 | 客户确认原因枚举后加入。 |
| 超时提醒 | 客户确认SLA和提醒对象后加入。 |
| 简版报销关联 | 只做工单关联和附件，不做完整审批/OCR。 |

### P2 仅预留字段

| 范围 | 说明 |
|---|---|
| 报价编号/报价状态 | 只记录编号、状态、负责人、附件。 |
| PO编号/ML编号 | 只做字段预留和查询。 |
| 采购到货状态 | 只记录预计/实际到货和状态。 |
| 客户自助查询编号 | 只预留编号，不开放客户门户。 |

## 3. 一期不包含范围

1. 完整报价系统。
2. 完整采购/仓库系统。
3. 完整报销审批/OCR。
4. WhatsApp自动抓取。
5. 电话语音转写。
6. 客户门户。
7. 实时地图监控。
8. HR/薪酬/绩效。
9. 完整ERP/CRM集成。
10. AI自动判断和自动派单。

## 4. 二期/三期规划

| 阶段 | 规划 |
|---|---|
| 二期 | 报价管理、采购管理、报销电子化、版本控制、更多看板、历史数据深化导入。 |
| 三期 | WhatsApp自动抓取、语音转写、OCR识别、客户自助查询、AI照片质量检查、知识库/专家推荐。 |

## 5. 客户需要提供的资料

| 资料 | 用途 |
|---|---|
| 客户/地址/屋苑清单 | 建立客户和地址主数据。 |
| 区域码与Team Leader/师傅映射 | 自动推荐Leader和派单。 |
| PM/CM真实工单样例 | 确认分类和字段。 |
| Daily Service Call Form样例 | 确认建单字段。 |
| Service Report / Job Completion样例 | 确认师傅提交和完工字段。 |
| 不可拍照客户/项目清单 | 配置照片豁免。 |
| 编号规则 | 普通工单、政府工单、项目编号、PO、ML、S/R。 |
| 钉钉账号、版本、宜搭许可、存储容量信息 | 确认技术可行性。 |

## 6. 待确认问题

1. PM/CM具体定义和边界。
2. 五个团队/区域分派规则。
3. 政府工单前缀规则。
4. 是否强制Leader确认完工。
5. 不可拍照客户清单。
6. P1是否纳入一期。
7. 超时提醒SLA和提醒对象。
8. 报销是否只做试点人员。
9. 图片在线保存期限和归档策略。
10. 钉钉/宜搭版本、账号数量和容量。

详见 [08-open-questions.md](08-open-questions.md)。

## 7. POC成功标准

| 标准 | 目标 |
|---|---|
| 试点范围 | 1个区域Team Leader、3-5名师傅、1-2名CS。 |
| 工单数量 | 20-30张真实或模拟工单。 |
| 移动端效率 | 师傅可独立完成接单、到场、拍照、提交。 |
| 照片上传 | 上传成功率不低于95%，不可拍照流程可正确豁免。 |
| 状态透明 | 管理层能看到每张未完成工单卡点。 |
| 权限 | 师傅不能看到非授权工单。 |
| 性能 | 手机端列表和详情加载可接受，无明显卡顿。 |

## 8. 变更管理原则

1. 本文件确认后，P0作为一期交付基线。
2. 未确认需求进入变更池，不直接加入一期。
3. P1需求只有在客户确认、工期允许、POC无重大风险时纳入一期。
4. P2字段预留不等于流程交付。
5. 新增模块如报价、采购、报销、客户门户、AI自动化，应单独评估工作量、费用和排期。
