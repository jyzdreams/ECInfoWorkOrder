# EC 工单系统项目宪法

## 核心原则

### I. 需求追溯原则
每个核心需求必须标注来源：客户沟通听记、官网背景，或“合理推断，需客户确认”。官网资料仅用于理解客户行业和服务类型，不得单独扩展为系统功能。没有资料支撑的功能不得写成已确认需求，也不得直接进入 MVP 或开发任务。

理由：EC 对范围、报价和交付边界敏感；需求追溯是避免范围膨胀和后续争议的基础。

### II. MVP 优先原则
第一版必须聚焦最小工单闭环：建单、受理、分派、处理、反馈、关闭、查询和基础统计。复杂 ITSM（IT Service Management，信息技术服务管理）、资产管理、知识库、SLA（Service Level Agreement，服务级别协议）计费、深度集成、AI（Artificial Intelligence，人工智能）自动化、完整 HR（Human Resources，人力资源）/仓库等能力不得进入 MVP，除非客户明确确认。

理由：当前阶段目标是让客户确认核心流程，而不是一次性建设大而全系统。

### III. 流程设计原则
后续每条流程必须明确适用场景、发起角色、参与角色、触发条件、状态变化、通知节点、异常处理、结束条件和验收标准。缺少任一要素的流程不得进入原型或开发任务拆分。

理由：EC 工单涉及 CS、CR、Team Leader、师傅、采购、财务和管理层等多方协作，流程不完整会导致责任不清和执行断点。

### IV. 页面原型原则
HTML 原型必须服务业务确认，体现真实流程、字段、状态、角色和操作边界。原型不追求复杂视觉设计，不做无依据功能，菜单、按钮、字段、状态和提示语默认使用中文。

理由：当前阶段是需求前置工作，原型的价值在于帮助客户确认业务，而不是替代正式开发。

### V. 状态、权限与一线可用性原则
工单状态、角色权限和页面操作按钮必须一致。每个按钮必须对应合法状态变化，每个状态变化必须有角色、条件和记录。面向 CS 和师傅的一线操作必须尽量简单，减少无必要必填项。

理由：客户资料显示复杂界面会降低一线录入意愿；状态和权限不一致会直接破坏流程闭环。

## 项目范围与边界

当前阶段只做需求梳理、可研评估、业务流程图、HTML 原型和后续开发任务拆分准备。不做真实后端开发、数据库实现、真实接口设计、生产级前端、上线部署或数据迁移。

第一阶段必须避免把项目扩大为通用 ITSM 平台。资产管理、知识库、SLA 计费、多系统深度集成、客户自助门户、自动客服、语音转写、WhatsApp 自动抓图、实时定位监控和未经确认的 AI 自动化均不属于当前 MVP。

## 需求发现与确认流程

后续工作必须先改需求文档，再改流程图，再改 HTML 原型，最后才进入开发任务拆分。任何新增功能必须标注来源或客户确认状态。资料中没有明确提到的内容，只能写为“合理推断，需客户确认”。

需求文档是流程图、原型、可研分析和任务拆分的源头。不得先做页面再反推需求，不得因为工具能力存在就把功能写入范围。

## 语言与交付规范

1. 所有正式交付文档默认使用简体中文。
2. Mermaid 流程图节点使用中文，必要时可保留英文缩写。
3. HTML 原型页面文案使用中文，包括菜单、按钮、字段、状态、提示语。
4. 技术术语首次出现必须解释中文含义。
5. 面向客户确认的内容优先使用业务语言。
6. 面向开发执行的内容可以使用必要技术术语，但必须说明业务含义。
7. 不得因为 spec-kit 默认模板为英文，就输出英文交付物。
8. 除非用户明确要求英文版本，否则不输出英文交付物。

## 治理

本项目宪法优先于临时功能想法、通用平台演示和未经确认的假设。需求规格、可研分析、流程图、原型和任务拆分必须通过本宪法检查后，才能作为正式交付物。

修订规则：
- 先改需求文档，明确新增或变更需求的来源。
- 再改流程图，确认角色、触发条件、状态流转、通知节点和验收标准。
- 再改 HTML 原型，更新页面、字段、按钮、状态和提示语。
- 最后才进入开发任务拆分。
- 任何新增功能必须标注来源或客户确认状态。

版本规则：
- 删除原则或不兼容治理变更为 MAJOR（主版本）。
- 新增原则或实质扩展义务为 MINOR（次版本）。
- 措辞澄清和错别字修正为 PATCH（补丁版本）。

合规检查必须验证：需求优先、需求追溯、MVP 清晰、流程五要素完整、状态权限一致、原型服务业务、中文交付规范。
